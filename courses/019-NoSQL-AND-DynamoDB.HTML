DynamoDB Architecture

A serverless NoSQL key-value + document database.

Quick Understanding

ConceptMeaningExam relevance
NoSQL DBaaSFully managed, serverless.No servers/patching.
Primary keyPK only, or PK + SK.Uniquely IDs items.
Capacity modesOn-demand vs Provisioned (RCU/WCU).Cost choice.
Regional + 3 AZsReplicated automatically.HA built-in.

Key Terms

TermSimple meaning
ItemA row (a record) in a table.
AttributeA field of an item.
Partition key (PK)The main key that decides where data is stored.
Sort key (SK)An optional second key to order items under a PK.

Explanation

graph TD
  TABLE[Table] --> ITEM[Items - max 400KB each]
  ITEM --> ATTR[Attributes]
  TABLE --> PK[Primary key: PK only OR PK + SK]
  TABLE --> CAP{Capacity}
  CAP --> OD[On-demand: pay per request]
  CAP --> PROV[Provisioned: set RCU/WCU]
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class TABLE a;
  • DynamoDB is a fully managed, serverless NoSQL database (key-value + document). Tables hold items (max 400KB) made of attributes.
  • Every item has a primary key: just a partition key (PK), or a PK + sort key (SK) (composite).
  • Capacity: On-demand (pay per request, auto-scales, for unpredictable load) or Provisioned (set RCU/WCU, cheaper for steady load).
  • It's a regional service, automatically replicated across 3 AZs (HA). Backups: on-demand + PITR (up to 35 days). Single-digit-ms latency.

Demo

Demo · CLI v2 (on-demand fits free tier)Situation: create a table + add/read items. Check first: CLI configured. Keep this table for L2–L4, L7.
Create an on-demand table (PK=species, SK=name)
aws dynamodb create-table --table-name a4l-animals \
  --attribute-definitions AttributeName=species,AttributeType=S \
                          AttributeName=name,AttributeType=S \
  --key-schema AttributeName=species,KeyType=HASH \
               AttributeName=name,KeyType=RANGE \
  --billing-mode PAY_PER_REQUEST
aws dynamodb wait table-exists --table-name a4l-animals
Expect: the table is created (HASH = partition key, RANGE = sort key).
Put + get an item
aws dynamodb put-item --table-name a4l-animals \
  --item '{"species":{"S":"cat"},"name":{"S":"whiskers"},"age":{"N":"3"}}'
aws dynamodb get-item --table-name a4l-animals \
  --key '{"species":{"S":"cat"},"name":{"S":"whiskers"}}'
Expect: the item back. The PK+SK (cat / whiskers) uniquely identifies it.
Fix / Key infoOn-demand (PAY_PER_REQUEST) needs no RCU/WCU planning. Keep the table for the next demos; cleanup is at the end of L7.
Clean up (after L7)
# aws dynamodb delete-table --table-name a4l-animals   # run after L7
Expect: leave it running for now.

Connects to NOW

Needed LATER
  • Serverless apps, session stores, global apps.

Exam Tip

  • Key = PK or PK + SK; item max 400KB.
  • Unpredictable spiky load → On-demand; steady load → Provisioned.

Operations, Consistency & Performance

Reads, writes, RCU/WCU and Query vs Scan.

Quick Understanding

ConceptMeaningExam relevance
QueryRead items by PK (+ SK condition).Efficient.
ScanRead the whole table, filter after.Expensive — avoid.
ConsistencyEventually (default) vs strongly consistent.Strong costs 2x.
RCU / WCURead/write capacity units.Calculations tested.

Key Terms

TermSimple meaning
RCURead Capacity Unit.
WCUWrite Capacity Unit.
Eventually consistentMay be slightly out of date for a moment.
Strongly consistentAlways the latest value (costs more).

Explanation

  • Reads: GetItem (one item), Query (items by PK, optional SK condition — efficient), Scan (reads everything, filters after — costly, avoid).
  • Consistency: eventually consistent (default, may be briefly stale, cheaper) vs strongly consistent (always latest, uses 2× the read capacity).
  • WCU: 1 WCU = 1 write/sec up to 1KB (round up). RCU: 1 RCU = 1 strongly-consistent read/sec up to 4KB, or 2 eventually-consistent reads of 4KB.
Quick RCU/WCU math [added for exam]
  • Write 10 items/sec, each 2.5KB → round to 3KB → 3 WCU each → 30 WCU.
  • Read 20 items/sec (strong), each 5KB → round to 8KB → 2 RCU each → 40 RCU. Eventually consistent → halve → 20 RCU.

Demo

Demo · CLI v2 (uses the L1 table)Situation: compare Query vs Scan. Check first: the a4l-animals table with an item.
Query by partition key (efficient)
aws dynamodb query --table-name a4l-animals \
  --key-condition-expression "species = :s" \
  --expression-attribute-values '{":s":{"S":"cat"}}'
Expect: only the items with species=cat (reads just that partition).
Scan the whole table (costly)
aws dynamodb scan --table-name a4l-animals
Expect: every item. Scan reads the entire table — avoid it at scale.
Fix / Key infoPrefer Query (uses the PK). Scan reads everything and burns capacity. Design keys so you can Query.

Build Knowledge

Connects to NOW
  • Indexes (L3) give more ways to Query; DAX (L6) caches reads.
Needed LATER
  • Cost-efficient data access design.

Exam Pointers

Exam Tip
  • Query = efficient (by PK); Scan = reads whole table (avoid).
  • Strongly consistent read = 2× the cost of eventually consistent.
  • WCU = 1KB/write; RCU = 4KB/strong-read (or 2 eventual).

Flashcards

Query vs Scan?tap
Query reads by PK (efficient); Scan reads the whole table (costly).
1 RCU equals?tap
1 strongly-consistent read of 4KB/sec (or 2 eventually-consistent).
1 WCU equals?tap
1 write of 1KB/sec.

LSI & GSI

Alternative ways to look at your data.

Quick Understanding

FeatureLSIGSI
KeysSame PK, different SKDifferent PK and SK
When createdAt table creation onlyAnytime
CapacityShares table'sIts own RCU/WCU
ConsistencyCan be strongEventual only

Key Terms

TermSimple meaning
IndexAn alternative view to query data differently.
ProjectionWhich attributes are copied into the index.
Base tableThe original table the index is built on.

Explanation

graph TD
  BASE[Base table - PK + SK] --> LSI[LSI: same PK, alternative SK]
  BASE --> GSI[GSI: different PK + SK]
  LSI --> LR[At creation only, shares capacity, can be strong]
  GSI --> GR[Anytime, own capacity, eventual only]
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class GSI a;
  • LSI = alternative sort key with the same partition key. Must be created with the table, shares the table's capacity, max 5, can be strongly consistent.
  • GSI = alternative partition key AND sort key. Created/deleted anytime, has its own capacity, max 20, eventually consistent only.
  • Default to GSI unless you specifically need the same PK or strong consistency (then LSI).

Demo

Demo · CLI v2 (uses the L1 table)Situation: add a GSI to query by age. Check first: the a4l-animals table.
Create a GSI on the age attribute
aws dynamodb update-table --table-name a4l-animals \
  --attribute-definitions AttributeName=age,AttributeType=N \
  --global-secondary-index-updates '[{"Create":{"IndexName":"age-index","KeySchema":[{"AttributeName":"age","KeyType":"HASH"}],"Projection":{"ProjectionType":"ALL"}}}]'
Expect: the GSI builds in the background. You can then Query by age (a new partition key). GSIs can be added anytime — LSIs cannot.
Fix / Key infoAn LSI must be defined at table creation; a GSI can be added later. GSIs only support eventually consistent reads.

Build Knowledge

Connects to NOW
  • More Query options (L2); avoids costly Scans.
Needed LATER
  • Access-pattern-driven data modelling.

Exam Pointers

Exam Tip
  • LSI = same PK, new SK, at creation only, shared capacity, can be strong.
  • GSI = new PK+SK, anytime, own capacity, eventual only.
  • "Need a new query after the table exists" → GSI.

Flashcards

LSI vs GSI keys?tap
LSI = same PK, different SK; GSI = different PK and SK.
Which can be added after creation?tap
GSI (LSI must be at creation).
Which supports strong consistency?tap
LSI (GSI is eventual only).

Mini Quiz

A table is live; you now need to query by a brand-new attribute as the key. Use?
A) LSIB) GSIC) Recreate the table
B. A GSI can be added anytime with a new partition key; LSI can't be added later.

React to item changes.

Quick Understanding

ConceptMeaningExam relevance
Stream24h time-ordered log of item changes.Per table.
View typesKEYS_ONLY / NEW / OLD / NEW_AND_OLD.What data appears.

Key Terms

TermSimple meaning
StreamA feed of changes (insert/modify/remove).
ImageThe item before (OLD) or after (NEW) a change.
TriggerCode that runs automatically on a change.

Explanation

graph LR
  CHANGE[Item insert/modify/remove] --> STREAM[Stream - 24h window]
  STREAM --> LAMBDA[Lambda trigger]
  LAMBDA --> ACT[Notify / aggregate / replicate]
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class STREAM a;
  • A stream is a 24-hour rolling, time-ordered log of item changes, enabled per table.
  • View types: KEYS_ONLY, NEW_IMAGE (after), OLD_IMAGE (before), NEW_AND_OLD_IMAGES (both).
  • Lambda can be triggered on new stream records — for notifications, aggregation, replication, or auditing.

Demo

Demo · CLI v2 (uses the L1 table)Situation: enable a stream. Check first: the a4l-animals table.
Enable the stream (new + old images)
aws dynamodb update-table --table-name a4l-animals \
  --stream-specification StreamEnabled=true,StreamViewType=NEW_AND_OLD_IMAGES
aws dynamodb describe-table --table-name a4l-animals \
  --query "Table.LatestStreamArn"
Expect: a stream ARN. In a real app you'd attach a Lambda (event source mapping) to react to changes.
Fix / Key infoStreams keep changes for 24h. NEW_AND_OLD_IMAGES shows the item before + after each change.

DynamoDB TTL

Auto-expire old items for free.

Quick Understanding

ConceptMeaningExam relevance
Per-item timestampAn attribute holds an expiry time.Epoch seconds.
Auto-deleteDynamoDB removes expired items.No WCU cost.
FreeNo extra charge.Cuts storage.

Key Terms

TermSimple meaning
TTLTime To Live — when an item should expire.
Epoch timeSeconds since 1970 (a number).

Explanation

  • TTL uses a per-item timestamp attribute (epoch seconds). After it passes, DynamoDB auto-deletes the item — without consuming WCU.
  • It's free and reduces stored data. Deletion usually happens within ~48h of expiry. [added for exam]
  • Expired-but-not-yet-deleted items can still appear in reads/scans (filter them); deletions also appear in Streams.

Demo

Demo · CLI v2 (uses the L1 table)Situation: enable TTL on an expiry attribute. Check first: the a4l-animals table.
Enable TTL
aws dynamodb update-time-to-live --table-name a4l-animals \
  --time-to-live-specification "Enabled=true,AttributeName=expiry"
Expect: TTL enabled on expiry. Items whose expiry (epoch seconds) is in the past get auto-deleted, no WCU cost.
Fix / Key infoTTL is free and uses no write capacity. Cleanup deadline reached — now delete the whole table.
Clean up (the whole section's table)
aws dynamodb delete-table --table-name a4l-animals
Expect: the table (with its GSI + stream) is deleted.

Connects to NOW

Needed LATER

Manual snapshotKept until you delete it.

Explanation

  • Traditional Redshift runs in a single AZ — an AZ failure is a risk.
  • Automated snapshots (to S3, scheduled, default ~1-day retention up to 35) + manual snapshots (kept until deleted) provide backup.
  • Cross-region snapshot copy gives DR; restoring a snapshot creates a new cluster (you can rebuild in another AZ/region).
  • Note: Multi-AZ is now available for some (RA3) clusters. [verify]
