AWS Accounts – The Basics

An account is a container — and a key SA tool.

Quick Understanding

ConceptMeaningExam relevance
AWS AccountA container that holds your identities and resources.Core idea — appears in many questions.
Account IDA unique 12-digit number for the account.Used in ARNs and cross-account access.
Account Root UserThe all-powerful first login (the account's email).Best practice: don't use it daily.
BoundaryAn account limits "blast radius" + holds its own bill.Why companies use many accounts.

Key Terms

TermSimple meaning
IdentityA login that can do things in the account (root user, IAM user, role).
ResourceSomething you create in AWS (a server, a storage bucket, a database).
Root userThe account owner login. Email + password. Cannot be limited.
Blast radiusHow much can break if something goes wrong. Smaller = safer.
ARNAmazon Resource Name — a unique ID/address for any AWS resource. [added for exam]

Explanation

graph TD
  ACC[AWS Account = container]
  ACC --> ROOT[Account Root User
full power, cannot be limited]
  ACC --> ID[12-digit Account ID]
  ACC --> IDN[Identities: IAM users / roles]
  ACC --> RES[Resources: EC2, S3, RDS...]
  ACC --> BILL[Its own bill]
  classDef r fill:#3a151c,stroke:#fb7185,color:#e7edf3;
  class ROOT r;
  • One account = one container with its own identities, resources and bill.
  • The root user logs in with the account email + password and has full, unlimited access.
  • An account is a security boundary: problems in one account don't spill into another.
  • An account is a billing boundary: easy to see cost per account/team/project.
Big idea for SAsCompanies use many accounts (e.g. dev, test, prod) to separate teams, limit blast radius, and split billing. You'll see this again in AWS Organizations.

Demo

Demo · CLI v2See your own account identity and account ID from the command line.

Situation: You manage Animals4life's account and want to confirm which account and identity your CLI is currently using before you build anything.

Check first: AWS CLI v2 installed and configured (you already did this). Run a quick version check if unsure.

Confirm CLI version
aws --version
Expect: a line starting aws-cli/2.x.x. If it says 1.x, upgrade to v2.
Show who you are + the account ID
aws sts get-caller-identity
Expect: JSON with UserId, Account (your 12-digit ID), and Arn (e.g. arn:aws:iam::1234..:user/iamadmin).
Fix / Key info
  • sts get-caller-identity is the safe "who am I?" command — it works for any identity and is great for troubleshooting "Access Denied" errors.
  • The Account field is your 12-digit account ID from the lesson theory.
Clean upNothing to delete — these commands only read info. No resources, no cost.

Build Knowledge

Connects to NOW

  • Lessons 6–8 secure the root user and add safer identities inside it.
Needed LATER

Exam Pointers

Exam Tip
  • The root user cannot be restricted by IAM policies inside its own account. Only an SCP (Organizations) can limit it.
  • Account = security boundary AND billing boundary. Remember both.
  • "Separate environments / limit blast radius / per-team billing" → answer is usually multiple accounts.
TrapDon't confuse Account Root User (the account owner login) with the Linux root user on an EC2 server. Same word, totally different thing.

Multi-Factor Authentication (MFA)

Add a second factor to protect the root user.

Quick Understanding

ConceptMeaningExam relevance
MFALogin needs password + a second proof (a code/device).Classic "secure the root user" answer.
Virtual MFAAn app on your phone makes 6-digit codes (TOTP).Most common MFA type.
FactorsKnowledge, possession, inherent (biometric), location.MFA = 2+ different factor types.

Key Terms

TermSimple meaning
FactorA type of proof you are you.
Knowledge factorSomething you know (password).
Possession factorSomething you have (phone app, security key).
TOTPTime-based One-Time Password — the 6-digit code that changes every 30s.
Authenticator appApp like Google Authenticator / Authy that generates TOTP codes.

Explanation

sequenceDiagram
  participant U as You
  participant AWS as AWS Login
  U->>AWS: 1. Email + password (know)
  AWS->>U: 2. Ask for MFA code
  U->>AWS: 3. 6-digit code from app (have)
  AWS->>U: 4. Access granted
  • Password alone = one factor. If stolen, the account is open.
  • MFA adds a second, different factor → a thief needs your password and your phone.
  • MFA device types: Virtual (app/TOTP), FIDO security key, hardware TOTP token.
  • Always enable MFA on the root user first. Then on important IAM users.

Demo

Part A · Root user MFA = console onlyRoot-user MFA cannot be set by CLI. Quick click-path:
Top-right account menu → Security credentials
Assign MFA → pick "Authenticator app" → name it
Scan the QR code in your authenticator app
Type 2 consecutive codes → Add MFA
Log out, log back in → now needs email + password + code
Part B · IAM-user MFA via CLI v2Situation: Animals4life wants MFA on an IAM user, scripted. Check first: you have a test IAM user (e.g. iamadmin) and admin rights.
Create a virtual MFA device + save the seed as a QR PNG
aws iam create-virtual-mfa-device \
  --virtual-mfa-device-name a4l-mfa \
  --outfile ./qr.png \
  --bootstrap-method QRCodePNG
Expect: JSON with a SerialNumber (an ARN), and a qr.png file you scan in your authenticator app.
Enable it on the user with 2 codes from the app
aws iam enable-mfa-device \
  --user-name iamadmin \
  --serial-number arn:aws:iam::ACCOUNT_ID:mfa/a4l-mfa \
  --authentication-code1 111111 \
  --authentication-code2 222222
Expect: no output = success. (Replace the two codes with real consecutive ones from your app.)
Fix / Key info
  • The SerialNumber for a virtual MFA is an ARN, not a hardware serial.
  • Codes must be two in a row so AWS can sync the clock (TOTP is time-based).
Clean upRemove the test MFA so nothing is left behind (MFA itself is free, but keep things tidy):
aws iam deactivate-mfa-device \
  --user-name iamadmin \
  --serial-number arn:aws:iam::ACCOUNT_ID:mfa/a4l-mfa

aws iam delete-virtual-mfa-device \
  --serial-number arn:aws:iam::ACCOUNT_ID:mfa/a4l-mfa

rm -f ./qr.png
Expect: no output from both AWS commands = deleted. Do not remove your real root/admin MFA.

IAM Basics

Make safe identities instead of using root.

Quick Understanding

ConceptMeaningExam relevance
IAMService to create identities + control what they can do.Very high — used in almost every section.
Global + freeIAM works across all regions and costs nothing.Common fact-check question.
Implicit denyNew identities can do nothing until granted.Core IAM rule.
PolicyA document that allows/denies actions.How permissions are granted.

Key Terms

TermSimple meaning
IAMIdentity and Access Management.
IAM userA long-term identity for one person or app.
IAM groupA folder of users so you give permissions once.
IAM roleAn identity you "borrow" temporarily (used by services/other accounts).
PolicyJSON rules: Allow/Deny which Actions on which Resources.
AuthenticationProving who you are (login).
AuthorizationWhat you are allowed to do after login.
Account aliasA friendly name in the IAM sign-in URL instead of the account ID.

Explanation

graph TD
  IAM[IAM - global, free] --> U[Users]
  IAM --> G[Groups]
  IAM --> R[Roles]
  P[Policies = Allow/Deny rules] -. attached to .-> U
  P -. attached to .-> G
  P -. attached to .-> R
  U --> NOTE[Start with NO permissions
implicit deny]
  classDef k fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class IAM k;
  • IAM is global (not per-region) and free.
  • New IAM identities start with zero permissions — everything is denied until you allow it.
  • Best practice: stop using root → create an IAM admin user + MFA for daily work.
  • IAM permissions can go up to (but not above) what root can do.
  • Two jobs: authentication (are you you?) and authorization (what can you do?).

Demo

Demo · CLI v2Create an admin IAM user the proper way: group → policy → user → console login. (The course does this in the console; here is the CLI equivalent.)

Situation: Animals4life needs an admin user iamadmin instead of using root. Check first: your CLI is configured with an identity that can manage IAM (run aws sts get-caller-identity).

(Optional) set a friendly account alias
aws iam create-account-alias --account-alias a4l-demo
Expect: no output. Sign-in URL becomes https://a4l-demo.signin.aws.amazon.com/console.
Create a group + attach AdministratorAccess
aws iam create-group --group-name admins

aws iam attach-group-policy \
  --group-name admins \
  --policy-arn arn:aws:iam::aws:policy/AdministratorAccess
Expect: create-group returns the group JSON; attach returns no output.
Create the user + put them in the group
aws iam create-user --user-name iamadmin

aws iam add-user-to-group \
  --group-name admins --user-name iamadmin
Expect: create-user returns user JSON (with the user ARN). add-user-to-group = no output.
Give console access (login profile)
aws iam create-login-profile \
  --user-name iamadmin \
  --password 'ChangeMe-Strong#2026' \
  --password-reset-required
Expect: JSON confirming the profile; user must change password at first sign-in.
Fix / Key info
  • Best practice: attach policies to a group, then add users — not to each user one by one.
  • AdministratorAccess is an AWS-managed policy (its ARN uses aws, not your account ID).
  • Real-world: add MFA to iamadmin right after (Lesson 6 Part B).
Clean upDelete everything so no stray identities remain (all IAM = free, but stay tidy):
aws iam delete-login-profile --user-name iamadmin
aws iam remove-user-from-group --group-name admins --user-name iamadmin
aws iam delete-user --user-name iamadmin
aws iam detach-group-policy --group-name admins \
  --policy-arn arn:aws:iam::aws:policy/AdministratorAccess
aws iam delete-group --group-name admins
aws iam delete-account-alias --account-alias a4l-demo
Expect: no output = each delete worked. Order matters: detach/remove links before deleting the user/group.

Build Knowledge

Connects to NOW
  • Replaces daily use of root (Lessons 4–5).
  • Pairs with MFA (Lesson 6) and Access Keys (Lesson 8).
Needed LATER
  • IAM policies, roles, ARNs, AWS Organizations, S3 bucket policies, EC2 instance roles — all build on this.

Exam Pointers

Exam Tip
  • IAM is global and free — remember both.
  • Default = implicit deny. No permission means denied.
  • Attach permissions to groups for scale, not to individual users.
  • Use roles (not access keys) when an AWS service needs permissions (e.g. EC2 → S3).
TrapAn IAM group is not an identity — you cannot log in "as a group" and a group cannot be a policy principal. It's just a container for users.

Global and free.

Nothing — implicit deny until a policy grants access.

To a group, then add users to the group.

Authentication = who you are (login). Authorization = what you may do.
Can an IAM group be logged into?tap
No. A group is just a container of users, not an identity.

  • Access keys = how CLI/API/SDK talk to AWS. The console uses username+password, not keys.
  • A key has 2 parts; the secret is shown only once — if lost, you must make a new key.
  • A user can have max 2 keys at a time (to rotate with no downtime).
  • You can deactivate a key (pause it) or delete it — you cannot edit it.
  • Root user can have access keys, but you should never create them (huge risk).

Demo

Demo · CLI v2 (you already have the CLI configured)You said you're comfortable installing/configuring the CLI — so this demo focuses on the exam-relevant key lifecycle: inspect, create a 2nd key, deactivate, delete.

Situation: You must rotate the access key for an Animals4life automation user safely. Check first: confirm your active identity and current keys.

See which credentials your CLI is using
aws configure list
aws sts get-caller-identity
Expect: configure list shows the access_key (masked) + profile/region; get-caller-identity shows your ARN + account.
List the existing keys for a user
aws iam list-access-keys --user-name iamadmin
Expect: JSON listing each AccessKeyId with Status = Active/Inactive and a create date. (No secret is ever shown here.)
Create a NEW (2nd) key — this is the only time you see the secret
aws iam create-access-key --user-name iamadmin
Expect: JSON with a new AccessKeyId AND SecretAccessKey. Copy the secret now — it is never shown again.
Deactivate the OLD key (pause without deleting)
aws iam update-access-key \
  --user-name iamadmin \
  --access-key-id AKIAOLDKEYEXAMPLE \
  --status Inactive
Expect: no output. Re-run list-access-keys → the old key now shows Inactive.
Delete the old key once the new one works
aws iam delete-access-key \
  --user-name iamadmin \
  --access-key-id AKIAOLDKEYEXAMPLE
Expect: no output = deleted. That is a clean, zero-downtime rotation (max 2 keys allowed = why this works).
Fix / Key info
  • The secret appears only at create-access-key time. Lost = recreate, never recover.
  • "Rotate" pattern: create new → switch apps → deactivate old → delete old.
  • Prefer IAM roles over long-term access keys whenever possible (temporary creds = safer).
Clean upIf you created a throwaway test key, delete it (see step above). Don't delete the key your CLI is actively using, or you'll lock yourself out.

Exam Pointers

Exam Tip
  • Access keys = CLI/API only. Console login does not use them.
  • Max 2 keys per user → enables safe rotation.
  • The secret is shown once. You can deactivate or delete, never edit or re-view.
  • "App on EC2 needs AWS access" → use a role, not stored access keys.
Easy to confuseAccess keys (long-term, CLI/API) vs console password (login) vs role/STS (temporary). Keep the three separate.
