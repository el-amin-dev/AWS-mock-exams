ACID vs BASE (+ CAP)

Two transaction philosophies.

Quick Understanding

ConceptMeaningExam relevance
ACIDStrong, consistent transactions.Relational/RDS.
BASEAvailable + eventually consistent.NoSQL/scale.
CAP theoremConsistency, Availability, Partition — pick 2.Trade-off framing.

Key Terms

TermSimple meaning
ACIDAtomic, Consistent, Isolated, Durable.
BASEBasically Available, Soft state, Eventually consistent.
Eventual consistencyAll copies agree after a short delay.

Explanation

graph TD
  ACID[ACID: strong consistency + transactions] --> REL[Relational / RDS]
  BASE[BASE: availability + eventual consistency] --> NOSQL[NoSQL / huge scale]
  CAP[CAP: in a network partition, choose Consistency OR Availability]
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class ACID a;
  • ACID guarantees reliable, consistent transactions — typical of relational databases; harder to scale massively.
  • BASE favours availability + eventual consistency — typical of NoSQL for huge scale.
  • CAP theorem: in a network partition you can keep either Consistency or Availability, not both.

Databases on EC2

You can — but should you?

Quick Understanding

ConceptMeaningExam relevance
Self-managed DBInstall/run a DB yourself on EC2.Max control.
You manage allPatching, backups, HA, scaling, replication.Big overhead.
When to chooseSpecial needs RDS can't meet.Rare; usually avoid.

Key Terms

TermSimple meaning
DBMSDatabase Management System (MySQL, Postgres, etc.).
MonolithOne server doing everything (app + DB together).
Admin overheadThe ongoing manual work to keep a DB healthy.

Explanation

Reasons FOR
  • Full OS + DB access and version control.
  • A feature/engine RDS doesn't offer.
  • Specific config/legacy requirement.
Reasons AGAINST
  • You handle patching, backups, HA, scaling, replication.
  • Must run 24/7; an AZ failure can take it down.
  • High admin overhead; no managed features.

Demo

Demo · on an EC2 instanceSituation: split the WordPress monolith — run MariaDB on EC2 and create the DB. Check first: a running instance (free-tier) you can SSH into.
Install + start MariaDB
sudo dnf install -y mariadb105-server
sudo systemctl enable --now mariadb
Expect: MariaDB installed and running.
Create a database + user for WordPress
sudo mysql -e "CREATE DATABASE a4lwordpress;
CREATE USER 'a4l'@'%' IDENTIFIED BY 'Password1234!';
GRANT ALL ON a4lwordpress.* TO 'a4l'@'%';
FLUSH PRIVILEGES;"
Expect: no errors — a database + user now exist for the app.
Fix / Key infoThis works, but now you own backups, patching and HA. That's exactly the pain RDS removes (L4). To move existing data you'd mysqldump from the old DB and import here.
Clean up
sudo mysql -e "DROP DATABASE a4lwordpress; DROP USER 'a4l'@'%';"
sudo systemctl disable --now mariadb
# terminate the EC2 instance when finished
Expect: database + user removed; stop the instance to avoid charges.

  • Default to RDS for managed reliability.

RDS Architecture

Managed database server as a service.

Quick Understanding

ConceptMeaningExam relevance
ManagedAWS runs patching, backups, HA options.Less overhead.
EnginesMySQL, MariaDB, PostgreSQL, Oracle, SQL Server (+ Aurora).Know the list.
In a VPCRuns in subnets via a DB subnet group; has an endpoint.Network detail.
No OS accessYou don't get the server (except RDS Custom).Key limit.

Key Terms

TermSimple meaning
DB instanceA managed database server you provision.
EndpointThe DNS name apps connect to.
DB subnet groupThe set of subnets RDS can place the DB in.

Explanation

graph TD
  APP[App] -->|endpoint DNS| RDS[RDS DB instance in VPC subnets]
  RDS --> STORE[EBS-backed storage]
  AWS[AWS manages: patching, backups, HA] --> RDS
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class RDS a;
  • RDS is a managed relational DB: AWS handles patching, backups and HA options; you don't get OS access (except RDS Custom, L9).
  • Engines: MySQL, MariaDB, PostgreSQL, Oracle, SQL Server, plus Aurora.
  • It runs inside a VPC using a DB subnet group and is reached via an endpoint (DNS).
  • Apps should connect via the endpoint name, not an IP (it can change on failover).

Demo

Time + costCreating an RDS instance takes ~10–15 min. db.t3.micro is free-tier eligible for 12 months; otherwise it bills hourly. Delete when finished.
Demo · CLI v2Situation: create a MySQL RDS instance for WordPress. Check first: default VPC (has a default DB subnet group).
Create the instance
RDSID=a4l-rds
aws rds create-db-instance --db-instance-identifier $RDSID \
  --db-instance-class db.t3.micro --engine mysql \
  --master-username admin --master-user-password 'Password1234!' \
  --allocated-storage 20 --backup-retention-period 7 \
  --no-multi-az --no-publicly-accessible
aws rds wait db-instance-available --db-instance-identifier $RDSID
Expect: the wait completes (~10–15 min) when the DB is available. Use a current engine version (e.g. MySQL 8.4.x) if asked.
Get the connection endpoint
aws rds describe-db-instances --db-instance-identifier $RDSID \
  --query "DBInstances[0].Endpoint.Address" --output text
Expect: a DNS endpoint like a4l-rds.xxxx.us-east-1.rds.amazonaws.com. WordPress would point at this and you'd mysqldump → import the data.
Fix / Key infoKeep $RDSID — Lessons 5–8 modify this instance. Apps use the endpoint, never a fixed IP.
Clean up (after L5–L8)
aws rds delete-db-instance --db-instance-identifier $RDSID \
  --skip-final-snapshot --delete-automated-backups
Expect: instance deleting — stops the charges.

Multi-AZ — Instance & Cluster

High availability (and a little read scaling).

Quick Understanding

ModeMeaningExam relevance
Multi-AZ Instance1 synchronous standby; NOT used for reads.Availability only.
Multi-AZ Cluster1 writer + 2 readers; readers usable.Some read scaling.
Same endpointFailover flips DNS to the standby.App keeps the name.

Key Terms

TermSimple meaning
SynchronousStandby is kept exactly in step with the primary.
StandbyA backup copy ready to take over.
FailoverSwitching to the standby when the primary fails.

Explanation

graph TD
  PRI[Primary AZ-a] -->|synchronous| STBY[Standby AZ-b - idle, no reads]
  PRI -->|fail| FO[Failover: DNS endpoint flips to standby]
  CL[Cluster mode: writer + 2 readers] --> RD[Readers CAN serve reads]
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class PRI a;
  • Multi-AZ Instance: one synchronous standby in another AZ. The standby is idle — it can't serve reads; it's purely for availability. Failover flips the same endpoint to the standby (~60–120s).
  • Multi-AZ Cluster: a writer + two readers across 3 AZs (synchronous). The readers can serve reads → limited read scaling, and faster failover.
  • Backups, patching and reboots use the standby/readers to reduce disruption.

Demo

CostMulti-AZ adds a standby → roughly doubles instance cost. Turn it back off after testing.
Demo · CLI v2Situation: turn Multi-AZ on, then off. Check first: the $RDSID instance from L4.
Enable Multi-AZ
aws rds modify-db-instance --db-instance-identifier $RDSID \
  --multi-az --apply-immediately
aws rds wait db-instance-available --db-instance-identifier $RDSID
aws rds describe-db-instances --db-instance-identifier $RDSID \
  --query "DBInstances[0].MultiAZ"
Expect: true — a standby now exists in another AZ.
Turn it back off (to stop the extra cost)
aws rds modify-db-instance --db-instance-identifier $RDSID \
  --no-multi-az --apply-immediately
Expect: Multi-AZ disabled again.
Fix / Key infoMulti-AZ Instance standby is NOT readable — don't use it for read scaling (that's read replicas, L7, or Multi-AZ Cluster readers).
Clean upLeft Multi-AZ off above; instance itself is cleaned up in L4's cleanup.

Build Knowledge

Connects to NOW
  • Contrast with read replicas (L7) for scaling.
Needed LATER
  • HA architectures, DR planning.

Exam Pointers

Exam Tip
  • Multi-AZ Instance standby = availability only, NOT readable.
  • Multi-AZ Cluster = writer + 2 readers; readers can serve reads.
  • Failover keeps the same endpoint (DNS flips).
  • Need read scaling → read replicas, not Multi-AZ Instance.

Flashcards

Can a Multi-AZ Instance standby serve reads?tap
No — it's for availability only.
Which Multi-AZ mode allows reads on replicas?tap
Multi-AZ Cluster (writer + 2 readers).
What happens to the endpoint on failover?tap
It stays the same; DNS flips to the standby.

Mini Quiz

B. Multi-AZ Instance gives synchronous standby + automatic failover (no read scaling).

Backups, Snapshots & Restore

Automatic PITR vs manual snapshots.

Quick Understanding

ConceptMeaningExam relevance
Automatic backupsDaily + 5-min transaction logs; retention 0–35 days.Point-in-time recovery.
Manual snapshotsYou take them; they persist past instance deletion.Long-term keep.
Restore = new instanceRestoring always creates a NEW DB.New endpoint.

Key Terms

TermSimple meaning
PITRPoint-in-time recovery — restore to a chosen moment.
RetentionHow many days automatic backups are kept (0–35).
Final snapshotAn optional snapshot taken when you delete a DB.

Explanation

graph TD
  AUTO[Automatic backups: daily + 5-min logs to S3] --> PITR[Point-in-time recovery 0-35 days]
  MAN[Manual snapshot] --> KEEP[Persists until you delete it]
  PITR --> NEW[Restore = NEW instance + new endpoint]
  MAN --> NEW
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class AUTO a;
  • Automatic backups: a daily backup plus transaction logs every 5 minutes to S3 → point-in-time recovery. Retention 0 (off) to 35 days; deleted with the instance unless you take a final snapshot.
  • Manual snapshots: taken by you; they live past instance deletion until you remove them.
  • Restoring (from snapshot or PITR) always creates a new instance with a new endpoint.

Demo

Demo · CLI v2Situation: take a manual snapshot and see restore. Check first: the $RDSID instance from L4 (available).
Create a manual snapshot
aws rds create-db-snapshot --db-instance-identifier $RDSID \
  --db-snapshot-identifier a4lwordpress-with-cat-post-mysql-8-4
aws rds wait db-snapshot-available \
  --db-snapshot-identifier a4lwordpress-with-cat-post-mysql-8-4
Expect: the wait completes when the snapshot is available.
Restore it to a NEW instance
aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier a4l-rds-restored \
  --db-snapshot-identifier a4lwordpress-with-cat-post-mysql-8-4 \
  --db-instance-class db.t3.micro
aws rds wait db-instance-available --db-instance-identifier a4l-rds-restored
Expect: a brand-new instance a4l-rds-restored with its own endpoint (the original is untouched).
Fix / Key infoRestore never overwrites — it builds a new DB. For PITR you'd use restore-db-instance-to-point-in-time within the retention window.
Clean up
aws rds delete-db-instance --db-instance-identifier a4l-rds-restored \
  --skip-final-snapshot --delete-automated-backups
aws rds delete-db-snapshot \
  --db-snapshot-identifier a4lwordpress-with-cat-post-mysql-8-4
Expect: restored instance + manual snapshot deleted.

Build Knowledge

Connects to NOW
  • Snapshots stored in S3 (S3 section); same idea as EBS snapshots.
Needed LATER
  • DR, cross-region snapshot copy, migration.

Exam Pointers

Exam Tip
  • Automatic backups: retention 0–35 days; 5-minute logs enable PITR; deleted with the instance (unless final snapshot).
  • Manual snapshots persist past deletion.
  • Restore = new instance + new endpoint (never in-place).

Flashcards

Max automatic backup retention?tap
35 days (0 disables it).
What enables point-in-time recovery?tap
Transaction logs to S3 every 5 minutes.
Does restoring overwrite the existing DB?tap
No — it creates a new instance with a new endpoint.

Read Replicas

Scale reads and improve recovery.

Quick Understanding

ConceptMeaningExam relevance
Read replicaAsync read-only copy; up to 5 per primary.Read scaling.
Same/cross regionCan live in another region.Global reads + DR.
Low RTOCan be promoted on failure.Recovery aid.
Not for corruptionCorruption replicates too.Common trap.

Key Terms

TermSimple meaning
AsynchronousReplica lags the primary by a little.
PromoteTurn a replica into a standalone writable DB.
RTORecovery Time Objective — how fast you recover.

Explanation

graph TD
  PRI[Primary - writes] -->|async| RR1[Read replica 1 - reads]
  PRI -->|async| RR2[Read replica - other region]
  RR1 -->|promote on failure| WRITE[Becomes standalone writer]
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class PRI a;
  • Read replicas are asynchronous, read-only copies — up to 5 per primary — with their own endpoints.
  • They give read scaling (offload read traffic) and can be in the same or another region (global reads + DR).
  • They support low-RTO recovery (promote a replica to a standalone DB).
  • They do not protect against data corruption — bad data replicates to them too.

Demo

CostA read replica is another running instance (extra cost). Delete it after.
Demo · CLI v2Situation: add a read replica. Check first: $RDSID available with backups enabled (retention > 0).
Create the read replica
aws rds create-db-instance-read-replica \
  --db-instance-identifier a4l-rds-rr \
  --source-db-instance-identifier $RDSID \
  --db-instance-class db.t3.micro
aws rds wait db-instance-available --db-instance-identifier a4l-rds-rr
Expect: a new read-only instance a4l-rds-rr with its own endpoint, replicating from the primary.
Fix / Key infoPoint read-heavy queries at the replica's endpoint. Remember: replicas don't fix corruption and are async (may lag slightly).
Clean up
aws rds delete-db-instance --db-instance-identifier a4l-rds-rr \
  --skip-final-snapshot
Expect: replica deleted (stops the extra charge).

Build Knowledge

Connects to NOW
  • Read scaling that Multi-AZ Instance (L5) doesn't provide.
Needed LATER
  • Global apps, DR, Aurora replicas.

Exam Pointers

Exam Tip
  • Read replicas = async, read scaling, up to 5, same/cross-region, can be promoted.
  • They don't help with data corruption (it replicates).
  • Multi-AZ = availability; read replicas = read scaling — don't confuse them.

Flashcards

How many read replicas per primary?tap
Up to 5 (direct).
Do read replicas fix data corruption?tap
No — corruption replicates to them.
Multi-AZ vs read replica purpose?tap
Multi-AZ = availability; read replicas = read scaling (+ some DR).

Mini Quiz

RDS Data Security

Encryption + IAM authentication.

Quick Understanding

ConceptMeaningExam relevance
Encryption at restKMS-based; set at creation.Can't add later directly.
In transitSSL/TLS to the DB.Protect connections.
IAM authLog in with IAM tokens, not a DB password.No stored DB creds.

Key Terms

TermSimple meaning
At restData stored on disk is encrypted.
IAM database authenticationUse a short-lived IAM token to connect.
TokenA temporary password generated from IAM (~15 min).

Explanation

graph TD
  KMS[KMS key] --> ENC[Encryption at rest - set at CREATE]
  ENC --> ALL[Covers storage, logs, snapshots, replicas]
  IAM[IAM auth] --> TOKEN[Short-lived token instead of DB password]
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class KMS a;
  • Encryption at rest uses KMS and must be chosen at creation. It can't be turned on directly later — snapshot, copy the snapshot with encryption, then restore.
  • Encryption covers storage, logs, snapshots and replicas.
  • In transit uses SSL/TLS.
  • IAM database authentication lets users connect with a short-lived IAM token instead of a stored DB password.

Demo

Demo · CLI v2Situation: create an encrypted instance + enable IAM auth. Check first: CLI configured. Note: another db.t3.micro — delete after (or reuse free tier).
Create an ENCRYPTED instance (encryption is set at create)
aws rds create-db-instance --db-instance-identifier a4l-rds-enc \
  --db-instance-class db.t3.micro --engine mysql \
  --master-username admin --master-user-password 'Password1234!' \
  --allocated-storage 20 --storage-encrypted \
  --enable-iam-database-authentication \
  --no-multi-az --no-publicly-accessible
aws rds wait db-instance-available --db-instance-identifier a4l-rds-enc
Expect: the wait completes; the instance is encrypted and IAM auth is on.
Confirm encryption + IAM auth
aws rds describe-db-instances --db-instance-identifier a4l-rds-enc \
  --query "DBInstances[0].{Enc:StorageEncrypted,IAM:IAMDatabaseAuthenticationEnabled}"
Expect: Enc: true and IAM: true.
Fix / Key infoYou can't encrypt an existing unencrypted instance in place — snapshot → copy with a KMS key → restore. IAM auth issues a ~15-minute token so apps don't store DB passwords.
Clean up
aws rds delete-db-instance --db-instance-identifier a4l-rds-enc \
  --skip-final-snapshot --delete-automated-backups
Expect: encrypted instance deleted.

Needed LATER
