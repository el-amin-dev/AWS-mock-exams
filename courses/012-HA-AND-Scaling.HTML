Evolution of the Elastic Load Balancer

From Classic to v2 (ALB + NLB).

Quick Understanding

TypeMeaningExam relevance
CLB (Classic)v1, 2009; basic; legacy.Avoid for new builds.
ALB (v2)Layer 7 HTTP/HTTPS; smart routing.Recommended for web.
NLB (v2)Layer 4 TCP/UDP; ultra-fast.High performance.
GWLBFor network appliances.Newest type.

Key Terms

TermSimple meaning
Layer 7Application layer — understands HTTP.
Layer 4Transport layer — just TCP/UDP, very fast.
v2The modern ELB generation (ALB/NLB).

Explanation

  • CLB (Classic, 2009) is legacy — one certificate per balancer, no smart routing. Avoid for new systems.
  • ALB (v2) works at Layer 7 (HTTP/HTTPS) with content-based routing + WAF.
  • NLB (v2) works at Layer 4 (TCP/UDP/TLS) for very high performance + static IPs.
  • GWLB is for running network appliances (L11). Use v2 for everything new.

ELB Architecture

Listeners, target groups and nodes.

Quick Understanding

ConceptMeaningExam relevance
ListenerPort/protocol the LB accepts (e.g. HTTPS 443).Front door.
Target groupThe backends (EC2/IP/Lambda) + health checks.Where traffic goes.
Nodes per AZLB places a node in each enabled subnet.Needs subnets.
Cross-zoneSpread across all targets in all AZs.ALB on; NLB off by default.

Key Terms

TermSimple meaning
ListenerA rule for what the LB listens on and how to route.
Target groupA named set of backends with health checks.
Internet-facing / internalPublic-facing vs private-only LB.
Cross-zone LBEven spread of traffic across AZs.

Explanation

graph TD
  DNS[ELB DNS name - Alias] --> NODE[LB nodes - one per enabled AZ subnet]
  NODE --> LISTEN[Listener: port + protocol]
  LISTEN --> TG[Target group + health checks]
  TG --> T1[Targets: EC2 / IP / Lambda]
  classDef a fill:#0e2647,stroke:#60a5fa,color:#e7edf3;
  class NODE a;
  • An ELB has a DNS name (use the Alias, never an IP for ALB) and places a node in each enabled AZ subnet (needs spare IPs, ideally a /27+).
  • A listener defines the port/protocol; it sends traffic to a target group (backends + health checks).
  • Targets can be EC2, IPs, or Lambda; can be internet-facing or internal.
  • Cross-zone load balancing spreads evenly across all AZs (ALB on by default; NLB off by default).

ALB vs NLB

Layer 7 features vs Layer 4 speed.

Quick Understanding

FeatureALB (Layer 7)NLB (Layer 4)
ProtocolsHTTP / HTTPSTCP / UDP / TLS
RoutingHost / path / headerNone (just forwards)
SpeedFastUltra-fast, ms latency
Static IPNoYes (EIP)
WAFYesNo

Key Terms

TermSimple meaning
Content-based routingRoute by URL path, hostname or headers.
WAFWeb firewall that filters HTTP attacks.
PrivateLinkPrivate service access — uses NLB.

Explanation

graph TD
  Q{Need HTTP features or extreme speed?} -->|HTTP routing, WAF, web app| ALB[ALB - Layer 7]
  Q -->|TCP/UDP, static IP, millions rps, PrivateLink| NLB[NLB - Layer 4]
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  classDef b fill:#0e2647,stroke:#60a5fa,color:#e7edf3;
  class ALB a; class NLB b;
  • ALB (Layer 7): HTTP/HTTPS, content-based routing (host/path/headers), WAF integration — best for web apps and microservices.
  • NLB (Layer 4): TCP/UDP/TLS, ultra-low latency, millions of requests/sec, static IP/EIP, preserves source IP — best for non-HTTP, gaming, and PrivateLink.
  • Need a static IP or extreme performance → NLB. Need HTTP routing/WAF → ALB.

Launch TemplateNewer, versioned, more features.Recommended.
Launch ConfigurationLegacy, no versions, immutable.Avoid for new.

Key Terms

TermSimple meaning
Launch Template (LT)A reusable, versioned definition of an instance.
Launch Configuration (LC)The old version; can't be edited (make a new one).
VersionLTs keep numbered versions you can roll back to.

Explanation

graph TD
  LT[Launch Template] --> WHAT[AMI + type + SG + key + user data + IAM role]
  WHAT --> ASG[Used by Auto Scaling Group]
  LT --> VER[Versioned + newer features]
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class LT a;
  • Both define WHAT gets launched: AMI, instance type, networking/SG, key pair, user data, IAM role, storage.
  • Launch Template is versioned, supports newer features, and is the recommended choice.
  • Launch Configuration is legacy and immutable (edit = create a new one).

Demo

Demo · CLI v2 (creating an LT is free)Situation: make a Launch Template for the ASG. Check first: CLI configured.
Create a Launch Template
AMI=$(aws ssm get-parameters \
  --names /aws/service/ami-amazon-linux-latest/al2023-ami-kernel-default-x86_64 \
  --query "Parameters[0].Value" --output text)
aws ec2 create-launch-template --launch-template-name a4l-lt \
  --launch-template-data "{\"ImageId\":\"$AMI\",\"InstanceType\":\"t3.micro\"}" \
  --query "LaunchTemplate.LaunchTemplateId" --output text
Expect: a launch-template ID like lt-0abc....
List its versions
aws ec2 describe-launch-template-versions --launch-template-name a4l-lt \
  --query "LaunchTemplateVersions[].VersionNumber"
Expect: version 1. Editing creates version 2, 3… (LCs can't do this).
Fix / Key infoKeep the LT — the ASG (L6) references it. Add user data, IAM role, SGs etc. as you build a real template.
Clean up (after L6–L9)
aws ec2 delete-launch-template --launch-template-name a4l-lt
Expect: template deleted (after the ASG is gone).

Keep the right number of healthy instances.

Quick Understanding

ConceptMeaningExam relevance
min / desired / maxFloor, target, ceiling for instance count.Core settings.
Self-healingReplaces unhealthy instances automatically.Resilience.
Spans AZsLaunches across the subnets you give it.HA design.
+ ELBRegisters instances with a target group.Scaling behind a LB.

Key Terms

TermSimple meaning
Desired capacityHow many instances the ASG aims to run.
Self-healingBad instance is terminated + replaced.
CooldownA pause after scaling before scaling again.

Explanation

graph TD
  ASG[Auto Scaling Group] --> MIN[min]
  ASG --> DES[desired]
  ASG --> MAX[max]
  ASG --> LT[uses Launch Template]
  ASG --> AZ[launches across subnets/AZs]
  ASG --> HEAL[replaces unhealthy instances]
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class ASG a;
  • An ASG keeps the desired number of instances between min and max, launching from a Launch Template.
  • It self-heals (replaces failed instances) and spreads across AZs for resilience.
  • It integrates with an ELB target group so new instances automatically receive traffic.
  • The ASG itself is free — you pay only for the instances.

Demo

Demo · CLI v2 (free-tier instances)Situation: create an ASG from the LT. Check first: the a4l-lt template (L5) + two subnet IDs.
Create the Auto Scaling group
VPC=$(aws ec2 describe-vpcs --filters Name=isDefault,Values=true \
  --query "Vpcs[0].VpcId" --output text)
SUBNETS=$(aws ec2 describe-subnets --filters Name=vpc-id,Values=$VPC \
  --query "Subnets[0:2].SubnetId" --output text | tr '\t' ',')
aws autoscaling create-auto-scaling-group --auto-scaling-group-name a4l-asg \
  --launch-template "LaunchTemplateName=a4l-lt,Version=\$Latest" \
  --min-size 1 --max-size 3 --desired-capacity 1 \
  --vpc-zone-identifier "$SUBNETS"
Expect: no output = created. It launches 1 t3.micro and will keep 1 running.
Watch it maintain capacity
aws autoscaling describe-auto-scaling-groups --auto-scaling-group-names a4l-asg \
  --query "AutoScalingGroups[0].Instances[].{Id:InstanceId,Health:HealthStatus}"
Expect: one healthy instance. If you terminate it, the ASG launches a replacement (self-healing).
Fix / Key infoKeep the ASG for L7–L9. The ASG decides how many/where; the LT decides what.
Clean up (after L9)
aws autoscaling delete-auto-scaling-group --auto-scaling-group-name a4l-asg --force-delete
Expect: ASG + its instances terminated.

Quick Understanding

PolicyMeaningExam relevance
Manual / ScheduledSet capacity by hand or on a timetable.Predictable patterns.
Target trackingKeep a metric at a target (e.g. CPU 50%).Simplest, recommended.
StepAdjust by amount based on breach size.Bigger breach = bigger step.

Key Terms

TermSimple meaning
Target trackingASG adds/removes to hold a metric near a target.
Step scalingDifferent size changes for different breach levels.
ScheduledScale at set times (e.g. business hours).

Explanation

graph TD
  P{Scaling policy} --> TT[Target tracking: hold CPU at 50%]
  P --> STEP[Step: bigger breach → bigger change]
  P --> SIMPLE[Simple: one change per alarm]
  P --> SCHED[Scheduled: scale at set times]
  classDef a fill:#0e2647,stroke:#60a5fa,color:#e7edf3;
  class TT a;
  • Target tracking keeps a metric at a target (e.g. average CPU 50%) — the simplest and recommended dynamic policy.
  • Step scaling changes capacity by amounts that depend on how big the alarm breach is.
  • Simple scaling makes one adjustment per alarm; scheduled scaling acts at set times for predictable patterns.

Demo

Demo · CLI v2 (policy is free to define)Situation: add a target-tracking policy. Check first: the a4l-asg from L6.
Keep average CPU at 50%
aws autoscaling put-scaling-policy --auto-scaling-group-name a4l-asg \
  --policy-name cpu50 --policy-type TargetTrackingScaling \
  --target-tracking-configuration '{"PredefinedMetricSpecification":{"PredefinedMetricType":"ASGAverageCPUUtilization"},"TargetValue":50.0}'
Expect: a policy ARN. The ASG now scales out/in to hold CPU near 50%.
Fix / Key infoTarget tracking is usually the right default. Use step when you need different responses to different breach sizes.
Clean up
aws autoscaling delete-policy --auto-scaling-group-name a4l-asg --policy-name cpu50
Expect: policy removed (ASG cleanup is in L6).

ASG Lifecycle Hooks

Pause instances for custom actions.

Quick Understanding

ConceptMeaningExam relevance
Lifecycle hookPause an instance as it launches or terminates.Custom bootstrap/cleanup.
Wait stateInstance waits until you signal or it times out.Default 1 hour.
Complete actioncomplete-lifecycle-action to continue.How you resume.

Key Terms

TermSimple meaning
Pending:WaitPaused while launching.
Terminating:WaitPaused while terminating.
Heartbeat timeoutHow long it waits before auto-continuing.

Explanation

graph TD
  LAUNCH[Instance launching] --> WAIT[Pending:Wait - paused]
  WAIT --> ACT[Run custom action: install/register/copy logs]
  ACT --> DONE[complete-lifecycle-action OR timeout]
  DONE --> INSERVICE[InService]
  classDef a fill:#0e2647,stroke:#60a5fa,color:#e7edf3;
  class WAIT a;
  • A lifecycle hook pauses an instance at launch (Pending:Wait) or terminate (Terminating:Wait) so you can run a custom action.
  • The instance stays in a wait state until you call complete-lifecycle-action or the timeout (default 1 hour) passes.
  • Used for tasks like finishing bootstrap, registering, or draining/saving logs before termination.

Demo

Demo · CLI v2 (hook is free to define)Situation: add a launch lifecycle hook. Check first: the a4l-asg from L6.
Add a launch hook (300s wait)
aws autoscaling put-lifecycle-hook --auto-scaling-group-name a4l-asg \
  --lifecycle-hook-name a4l-launch-hook \
  --lifecycle-transition autoscaling:EC2_INSTANCE_LAUNCHING \
  --heartbeat-timeout 300 --default-result CONTINUE
Expect: hook created. New instances pause in Pending:Wait for up to 300s (or until you complete the action).
Fix / Key infoResume early with complete-lifecycle-action. --default-result decides what happens on timeout (CONTINUE or ABANDON).
Clean up
aws autoscaling delete-lifecycle-hook --auto-scaling-group-name a4l-asg \
  --lifecycle-hook-name a4l-launch-hook
Expect: hook removed.

Health Checks: EC2 vs ELB

Detect failures at the right level.

Quick Understanding

TypeMeaningExam relevance
EC2 (default)Hardware/software status checks only.Misses app failures.
ELBApp-level (HTTP response) checks.Catches app problems.
CustomYou report health yourself.Special cases.

Key Terms

TermSimple meaning
EC2 status checkIs the instance/host basically running?
ELB health checkDoes the app respond correctly (e.g. HTTP 200)?
Grace periodTime after launch before health checks count.

Explanation

graph TD
  HC{ASG health check type} --> EC2[EC2: instance running? - default]
  HC --> ELB[ELB: app responds? - deeper]
  EC2 --> MISS[A hung app still passes EC2 checks]
  ELB --> CATCH[ELB check catches the broken app]
  classDef a fill:#0e2647,stroke:#60a5fa,color:#e7edf3;
  class ELB a;
  • EC2 status checks (the ASG default) only confirm the instance/host is running — a hung or broken app can still "pass".
  • ELB health checks test the application (e.g. HTTP 200) and catch app-level failures — better for web tiers (enable them).
  • A grace period gives new instances time to boot before checks count.

Demo

Demo · CLI v2Situation: switch the ASG to use ELB health checks. Check first: the a4l-asg from L6 (an ELB target group is needed to actually use ELB checks).
Set health-check type to ELB
aws autoscaling update-auto-scaling-group --auto-scaling-group-name a4l-asg \
  --health-check-type ELB --health-check-grace-period 120
aws autoscaling describe-auto-scaling-groups --auto-scaling-group-names a4l-asg \
  --query "AutoScalingGroups[0].HealthCheckType"
Expect: ELB. With an ELB attached, the ASG now replaces instances that fail the app-level check, not just hardware checks.
Fix / Key infoFor web apps behind an ELB, use ELB health checks so the ASG removes instances whose app is broken even if the host is "healthy".
Clean upNow tear down the chain: ASG (L6 cleanup) then the launch template (L5 cleanup).

SSL Offload & Session Stickiness

How ELBs handle HTTPS, and keeping a user on one server.

Quick Understanding

ConceptMeaningExam relevance
SSL BridgingELB decrypts, inspects, re-encrypts to backend.Default ALB HTTPS.
SSL Pass-throughEncrypted traffic goes straight to the backend.NLB; ELB can't read it.
SSL OffloadELB decrypts, sends plaintext to backend.Less backend CPU.
StickinessSame user → same target via a cookie.Stateful apps.

Key Terms

TermSimple meaning
SSL/TLSEncryption for data in transit (HTTPS).
CertificateProves identity + enables encryption.
Session stickinessA user keeps hitting the same backend.
WPHOMEWordPress setting storing its own URL (can be hardcoded wrong).

Explanation

graph TD
  C[Client HTTPS] --> ELB[ELB]
  ELB -->|Bridging: decrypt + inspect + re-encrypt| B1[Backend HTTPS]
  ELB -->|Offload: decrypt then plaintext| B2[Backend HTTP]
  ELB -.Pass-through: stays encrypted, ELB cannot read.-> B3[Backend decrypts]
  classDef a fill:#0e2647,stroke:#60a5fa,color:#e7edf3;
  class ELB a;
  • SSL Bridging: ELB decrypts, can inspect, then re-encrypts to the backend. Needs a certificate on the ELB and backend (default for ALB HTTPS).
  • SSL Pass-through: traffic stays encrypted to the backend (NLB/TCP); the ELB can't read it; cert only on the backend.
  • SSL Offload: ELB decrypts and sends plaintext HTTP to the backend; cert only on the ELB — saves backend CPU but the internal hop is unencrypted.
  • Session stickiness: an ALB cookie sends a user back to the same target. Useful for stateful apps, but it can unbalance load — better to externalize sessions (e.g. DynamoDB/ElastiCache).

Demo overview · Architecture Evolution (6 stages)Situation: evolve Animals4life WordPress from one manual box into a scalable, resilient design. Each stage reuses skills from earlier sections. (This is a multi-hour build with billable ALB/RDS/EFS — follow it stage-by-stage and clean up at Stage 6.)
