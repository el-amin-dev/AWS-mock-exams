Introduction to Containers

The parts that matter for AWS + the exam.

Quick Understanding

ConceptMeaningExam relevance
Container vs VMContainer shares the host OS kernel; VM virtualizes full hardware + OS.Lighter, denser, faster.
Image = layersRead-only template built from stacked filesystem layers.Shared/cached → efficient.
RegistryStores and shares images (Docker Hub, ECR).ECR is the AWS one.

Key Terms

TermSimple meaning
ImageA read-only template that a container runs from.
LayerOne step of the image's filesystem; layers stack and are reused.
ContainerA running instance of an image (adds a writable layer).
RegistryA store for images you push/pull.

Explanation

graph TD
  HOST[Host OS + container engine] --> C1[Container A]
  HOST --> C2[Container B]
  IMG[Image = read-only layers] -->|run| C1
  IMG -->|run| C2
  REG[(Registry: Docker Hub / ECR)] -->|pull| IMG
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class IMG a;
You know Docker — the AWS-relevant points
  • Containers share the host kernel → far lighter than VMs, so you fit more per host and start in seconds.
  • Images are immutable, layered; shared base layers save space and speed pulls. Keep images small.
  • A container should run one main process, expose ports, and store state outside the container (volumes, S3, RDS).
  • For ECS you'll wrap the image in a task definition (next lesson); the image lives in a registry (ECR, L4).

ECS — Concepts

Cluster, container def, task def, service def.

Quick Understanding

ConceptMeaningExam relevance
Container definitionWhich image + which ports.Smallest piece.
Task definitionBlueprint: containers + CPU/memory + roles + network.Includes task role.
TaskA running copy of a task definition.Not HA alone.
ServiceKeeps N tasks running, scales, links a load balancer.Gives HA + scaling.

Key Terms

TermSimple meaning
ClusterWhere ECS runs your tasks/services.
Task roleIAM role the container app uses to call AWS.
Execution roleIAM role ECS uses to pull images + send logs.
Desired countHow many tasks a service should keep running.

Explanation

graph TD
  CD[Container definition: image + ports] --> TD[Task definition: containers + CPU/mem + task role]
  TD --> TASK[Task = running copy]
  TD --> SVC[Service: keeps N tasks, scales, + load balancer]
  SVC --> CLUSTER[Cluster runs it all]
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class TD a;
  • Container definition = the image + ports for one container.
  • Task definition = the full blueprint: one or more containers, CPU/memory, task role (app permissions), network mode.
  • Task = a running copy of the task definition. On its own it is not scalable or highly available.
  • Service = keeps a desired count of tasks running, scales them, and connects a load balancer → this gives HA + scaling.
  • Task role (app permissions) is different from execution role (ECS pulling images / writing logs). [added for exam]

Demo

Demo · CLI v2 (free — just metadata)Situation: register a task definition, list it, then remove it. Check first: CLI configured. No containers run here (that's L3).
Write + register a simple task definition
cat > td.json <<'EOF'
{
  "family": "a4l-demo",
  "networkMode": "awsvpc",
  "requiresCompatibilities": ["FARGATE"],
  "cpu": "256",
  "memory": "512",
  "containerDefinitions": [
    { "name": "web", "image": "httpd:latest",
      "portMappings": [{ "containerPort": 80, "protocol": "tcp" }],
      "essential": true }
  ]
}
EOF
aws ecs register-task-definition --cli-input-json file://td.json \
  --query "taskDefinition.{Family:family,Rev:revision}"
Expect: family a4l-demo and revision 1 — a versioned blueprint now exists.
List task definitions
aws ecs list-task-definitions --family-prefix a4l-demo
Expect: the ARN of a4l-demo:1.
Fix / Key infoTask definitions are versioned (revisions). Editing creates a new revision; old ones stay until deregistered. Registering is free — you only pay when tasks run.
Clean up
aws ecs deregister-task-definition --task-definition a4l-demo:1
rm -f td.json
Expect: status becomes INACTIVE. (Fully running this app is the L3 demo.)

Connects to NOW

Needed LATER

Exam Pointers

Exam Tip
  • Task = one running copy (not HA). Service = keeps N tasks + scaling + load balancer (HA).
  • Task role = app's AWS permissions (best practice, not stored keys).
  • Task definitions are versioned (revisions).

ECS — Cluster Mode

EC2 mode vs Fargate mode.

Quick Understanding

ConceptMeaningExam relevance
EC2 modeECS puts EC2 instances in your account; you manage them.Pay for EC2 always.
Fargate modeServerless; shared infra; ENIs in your VPC.Pay per running task.
ChoosingControl/existing EC2 vs no admin/pay-per-use.Common scenario.

Key Terms

TermSimple meaning
FargateServerless containers — no servers for you to manage.
Launch typeEC2 or FARGATE — where the task runs.
ENI injectionFargate puts a network card for the task into your VPC.

Explanation

graph TD
  TD[Task definition] --> Q{Launch type?}
  Q -->|EC2| EC2[You run + pay for EC2 instances
more control, patch + scale them]
  Q -->|Fargate| FAR[Serverless: AWS runs it
pay per vCPU+memory while running]
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  classDef b fill:#0e2647,stroke:#60a5fa,color:#e7edf3;
  class EC2 a; class FAR b;
  • EC2 mode: ECS uses EC2 instances in your account. You manage/patch/scale them and pay for them even when idle. Good if you want control or already run lots of (reserved) EC2.
  • Fargate mode: serverless — no instances to manage. AWS runs tasks on shared infra and injects an ENI into your VPC. You pay only for the vCPU + memory while tasks run. Good for bursty, batch, or low-ops workloads.
  • Both use the same task/service definitions; only the launch type differs.

Demo

Cost warningA running Fargate task bills per second (vCPU+memory) — only cents for a few minutes, but stop the task and delete the cluster right after.
Demo · CLI v2 — "container of cats" on FargateSituation: deploy a public web container on Fargate and open it. Check first: a default VPC with a public subnet (or your VPC). Uses httpd as a safe stand-in — swap in the container-of-cats image if you have it.
Create a cluster
aws ecs create-cluster --cluster-name a4l-cats \
  --query "cluster.clusterName"
Expect: a4l-cats.
Register a Fargate task definition
cat > cats.json <<'EOF'
{
  "family": "a4l-cats",
  "networkMode": "awsvpc",
  "requiresCompatibilities": ["FARGATE"],
  "cpu": "256",
  "memory": "512",
  "containerDefinitions": [
    { "name": "cats", "image": "httpd:latest",
      "portMappings": [{ "containerPort": 80, "protocol": "tcp" }],
      "essential": true }
  ]
}
EOF
aws ecs register-task-definition --cli-input-json file://cats.json \
  --query "taskDefinition.taskDefinitionArn" --output text
Expect: a task-definition ARN ending a4l-cats:1.
Grab a public subnet + make a security group (port 80)
VPC=$(aws ec2 describe-vpcs --filters Name=isDefault,Values=true \
  --query "Vpcs[0].VpcId" --output text)
SUBNET=$(aws ec2 describe-subnets --filters Name=vpc-id,Values=$VPC \
  --query "Subnets[0].SubnetId" --output text)
SG=$(aws ec2 create-security-group --group-name a4l-cats-sg \
  --description "cats 80" --vpc-id $VPC --query GroupId --output text)
aws ec2 authorize-security-group-ingress --group-id $SG \
  --protocol tcp --port 80 --cidr 0.0.0.0/0
echo "subnet=$SUBNET sg=$SG"
Expect: a subnet ID + SG ID, with port 80 open.
Run the task on Fargate with a public IP
TASK=$(aws ecs run-task --cluster a4l-cats --launch-type FARGATE \
  --task-definition a4l-cats \
  --network-configuration "awsvpcConfiguration={subnets=[$SUBNET],securityGroups=[$SG],assignPublicIp=ENABLED}" \
  --query "tasks[0].taskArn" --output text)
aws ecs wait tasks-running --cluster a4l-cats --tasks $TASK
echo $TASK
Expect: a task ARN; the wait finishes when it's RUNNING (a Docker Hub public image needs the public IP to pull).
Find the task's public IP, then open it
ENI=$(aws ecs describe-tasks --cluster a4l-cats --tasks $TASK \
  --query "tasks[0].attachments[0].details[?name=='networkInterfaceId'].value" --output text)
aws ec2 describe-network-interfaces --network-interface-ids $ENI \
  --query "NetworkInterfaces[0].Association.PublicIp" --output text
Expect: a public IP; open http://THAT_IP → the web page loads (the cats app, or Apache's default page with the stand-in image).
Fix / Key infoFor a private ECR image or CloudWatch logs you'd also attach an execution role (ecsTaskExecutionRole). A public image with a public IP needs neither.
Clean up
aws ecs stop-task --cluster a4l-cats --task $TASK
aws ecs deregister-task-definition --task-definition a4l-cats:1
aws ecs delete-cluster --cluster a4l-cats
aws ec2 delete-security-group --group-id $SG
rm -f cats.json
Expect: task stopped (billing stops), cluster + SG removed. No EC2 to delete — that's the Fargate benefit.

Elastic Container Registry (ECR)

Where your container images live on AWS.

Quick Understanding

ConceptMeaningExam relevance
ECRAWS managed Docker image registry.The AWS image store.
Private vs publicPrivate (IAM-controlled) or public gallery.Access control.
ExtrasImage scanning, lifecycle policies, replication.Security + cost.

Key Terms

TermSimple meaning
RepositoryA named place that holds versions (tags) of one image.
ScanningChecking images for known security problems.
Lifecycle policyRules to auto-delete old images (save cost).

Explanation

graph TD
  DEV[Build image] -->|push| ECR[(ECR private repo)]
  ECR -->|IAM controls access| PULL[ECS / EKS pull image]
  ECR --> SCAN[Scan for vulnerabilities]
  ECR --> LIFE[Lifecycle policy deletes old images]
  classDef a fill:#0e2647,stroke:#60a5fa,color:#e7edf3;
  class ECR a;
  • ECR is a managed registry: private repos use IAM for permissions; there's also a public gallery.
  • It's per-region, stores images durably, and integrates with ECS/EKS (and the task execution role to pull private images).
  • Extras: image scanning (security), lifecycle policies (auto-clean old tags), and replication across regions/accounts.
  • You pay for storage + data transfer.

Demo

Demo · CLI v2Situation: create a private ECR repo and see the push commands. Check first: CLI configured (Docker only needed if you actually push).
Create a private repository
aws ecr create-repository --repository-name a4l-app \
  --image-scanning-configuration scanOnPush=true \
  --query "repository.repositoryUri" --output text
Expect: a URI like <acct>.dkr.ecr.<region>.amazonaws.com/a4l-app; scan-on-push is on.
(Optional) the push flow — needs Docker
ACCT=$(aws sts get-caller-identity --query Account --output text)
REGION=us-east-1
aws ecr get-login-password --region $REGION | \
  docker login --username AWS --password-stdin $ACCT.dkr.ecr.$REGION.amazonaws.com
# docker tag myimage:latest $ACCT.dkr.ecr.$REGION.amazonaws.com/a4l-app:latest
# docker push $ACCT.dkr.ecr.$REGION.amazonaws.com/a4l-app:latest
Expect: Login Succeeded. The tag/push lines (commented) upload an image if you have one built.
Fix / Key infoECS/EKS pull private ECR images using the task execution role — no passwords stored in the task. Access is all IAM.
Clean up
aws ecr delete-repository --repository-name a4l-app --force
Expect: repo (and any images in it) deleted.

Kubernetes 101

Open-source container orchestration (brief).

Quick Understanding

ConceptMeaningExam relevance
Kubernetes (K8s)Open-source system to deploy/scale/manage containers.Cloud-agnostic.
ClusterControl plane (brain) + worker nodes (run containers).Core structure.
PodSmallest unit; one or more containers together.K8s vocab.
PortableRuns on-prem, in any cloud, or AWS (EKS).Avoids lock-in.

Key Terms

TermSimple meaning
Control planeThe "brain" that schedules and manages everything.
NodeA worker machine that runs the containers.
PodThe smallest deployable unit (one+ containers sharing network).
kubectlThe command-line tool to talk to a cluster.

Explanation

graph TD
  CP[Control plane - schedules + manages] --> N1[Node 1]
  CP --> N2[Node 2]

  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;

  • Kubernetes is an open-source orchestrator: it deploys, scales and heals containers — similar job to ECS, but portable across clouds and on-prem.
  • A cluster = a control plane (the brain) + worker nodes that run pods.
  • A pod is the smallest unit — one or more containers that share network/storage.
  • The AWS managed version is EKS (next lesson).

Elastic Kubernetes Service (EKS) 101

AWS-managed Kubernetes.

Quick Understanding

ConceptMeaningExam relevance
EKSManaged Kubernetes; AWS runs the control plane.K8s without the admin.
Compute optionsWorker nodes on EC2 (self/managed) or Fargate.Like ECS choices.
Control plane costYou pay an hourly fee for the control plane.Cost trap.
Use when…You specifically want Kubernetes / portability.ECS vs EKS choice.

Key Terms

TermSimple meaning
Managed control planeAWS runs + scales the K8s brain across AZs.
Managed node groupEC2 worker nodes AWS helps you manage.
EKS on FargateRun pods serverless, no nodes to manage.

Explanation

graph TD
  EKS[EKS managed control plane - HA across AZs] --> COMP{Worker compute?}
  COMP -->|EC2| NG[Self-managed or managed node groups]
  COMP -->|Fargate| FG[Serverless pods]
  EKS --> VPC[Runs inside your VPC + IAM]
  classDef a fill:#0e2647,stroke:#60a5fa,color:#e7edf3;
  class EKS a;
  • EKS = managed Kubernetes: AWS runs the control plane (HA across AZs); you run workloads.
  • Worker compute can be EC2 (self-managed or managed node groups) or Fargate (serverless pods).
  • It runs inside your VPC and integrates with IAM.
  • You pay an hourly control-plane fee plus the EC2/Fargate compute. Pick EKS when you specifically need Kubernetes or portability; otherwise ECS is simpler. [added for exam]
