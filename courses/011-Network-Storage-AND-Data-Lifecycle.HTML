EFS Architecture

One file system, many EC2 instances.

Quick Understanding

ConceptMeaningExam relevance
EFS = managed NFSShared Linux file system for many EC2 at once.File storage, not block/object.
Mount targetsAn ENI per AZ that instances mount through.VPC + per-AZ access.
Enables scalingMany instances share the same files.Horizontal scaling.
Storage classesStandard / One Zone (+ IA), lifecycle to IA.Cost control.

Key Terms

TermSimple meaning
NFSNetwork File System — a way to share files over a network (Linux).
Mount targetA network entry point (ENI + IP) for EFS in an AZ's subnet.
POSIXThe Linux file permission model EFS uses.
IAInfrequent Access — cheaper storage for rarely-used files.

Explanation

graph TD
  EFS[EFS file system] --> MTA[Mount target AZ-a]
  EFS --> MTB[Mount target AZ-b]
  MTA --> E1[EC2 in AZ-a]
  MTB --> E2[EC2 in AZ-b]
  SG[Security group on mount targets allows NFS 2049] --> MTA
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class EFS a;
  • EFS is a managed NFS file system for Linux EC2 (POSIX permissions). Many instances mount it at the same time — unlike EBS, which is usually one instance.
  • It lives in a VPC; instances connect through mount targets (an ENI + IP in each AZ). A security group on the mount targets must allow NFS (port 2049).
  • Standard is multi-AZ resilient; One Zone is cheaper but single-AZ. Lifecycle moves unused files to IA to save cost.
  • Performance: General Purpose (default, low latency) vs Max I/O (higher throughput, more latency). Throughput: Bursting / Elastic / Provisioned. [added for exam]
  • Classic use: shared content like WordPress wp-content, so multiple WordPress instances can run together.

Demo

Small costEFS bills per GB stored (tiny for a few test files) + any provisioned throughput. Delete the file system after.
Demo · CLI v2 + on instancesSituation: create EFS, add mount targets, mount on two instances, then move WordPress wp-content onto it. Check first: a VPC with subnets in 2 AZs and two running instances.
Create the file system
EFS=$(aws efs create-file-system --performance-mode generalPurpose \
  --throughput-mode elastic --encrypted \
  --tags Key=Name,Value=a4l-efs --query "FileSystemId" --output text)
aws efs describe-file-systems --file-system-id $EFS \
  --query "FileSystems[0].LifeCycleState"
Expect: a file-system ID like fs-0abc..., state becoming available.
Security group allowing NFS (2049)
VPC=$(aws ec2 describe-vpcs --filters Name=isDefault,Values=true \
  --query "Vpcs[0].VpcId" --output text)
SG=$(aws ec2 create-security-group --group-name a4l-efs-sg \
  --description "efs nfs" --vpc-id $VPC --query GroupId --output text)
aws ec2 authorize-security-group-ingress --group-id $SG \
  --protocol tcp --port 2049 --cidr 0.0.0.0/0   # demo only - tighten to instance SG
Expect: an SG ID with port 2049 open (in real life allow only the instances' SG).
Create a mount target in each AZ subnet
for SUBNET in $(aws ec2 describe-subnets --filters Name=vpc-id,Values=$VPC \
  --query "Subnets[].SubnetId" --output text); do
  aws efs create-mount-target --file-system-id $EFS \
    --subnet-id $SUBNET --security-groups $SG 2>/dev/null
done
aws efs describe-mount-targets --file-system-id $EFS \
  --query "MountTargets[].{AZ:AvailabilityZoneName,State:LifeCycleState}"
Expect: a mount target per AZ, each becoming available.
On each EC2 instance: install + mount
sudo dnf install -y amazon-efs-utils
sudo mkdir -p /efs
sudo mount -t efs -o tls FS_ID:/ /efs     # replace FS_ID with your $EFS
df -h | grep efs
Expect: /efs mounted (encrypted in transit via TLS). A file written on one instance appears on the other.
Move WordPress wp-content onto EFS
sudo mv /var/www/html/wp-content/* /efs/ 2>/dev/null
sudo umount /efs && sudo mount -t efs -o tls FS_ID:/ /var/www/html/wp-content
Expect: wp-content now lives on shared EFS, so several WordPress instances can serve the same media → horizontal scaling.
Fix / Key infoEFS is Linux/NFS only. For Windows file shares use FSx for Windows; for block storage use EBS; for objects use S3.
Clean up
# on instances: sudo umount /efs /var/www/html/wp-content
for MT in $(aws efs describe-mount-targets --file-system-id $EFS \
  --query "MountTargets[].MountTargetId" --output text); do
  aws efs delete-mount-target --mount-target-id $MT; done
sleep 30
aws efs delete-file-system --file-system-id $EFS
aws ec2 delete-security-group --group-id $SG
Expect: mount targets, file system and SG removed.

Exam Pointers

Exam Tip
  • EFS = managed NFS, Linux only, shared by many EC2 at once (file storage).
  • Accessed via mount targets (per AZ) over NFS port 2049; SG controls access.
  • Windows shares → FSx for Windows, not EFS.
  • Use EFS to let multiple instances share files (e.g. WordPress wp-content) for scaling.

AWS Backup

Central, policy-based data protection.

Quick Understanding

ConceptMeaningExam relevance
Central backupOne place to back up many services.EBS, RDS, EFS, DynamoDB, S3...
Backup planRules: frequency, window, lifecycle, retention.Policy-based.
Backup vaultStores recovery points; can be KMS-encrypted.Vault Lock = WORM.
Org-wideDeploy policies across accounts via Organizations.Governance.

Key Terms

TermSimple meaning
Recovery pointOne backup you can restore from.
Backup planThe schedule + rules for taking backups.
Vault LockMakes backups immutable (can't delete) for compliance.
Resource assignmentWhich resources a plan backs up (by tag/ID).

Explanation

graph TD
  PLAN[Backup plan: schedule + lifecycle + retention] --> ASSIGN[Assign resources by tag/ID]
  ASSIGN --> JOB[Backup jobs run]
  JOB --> VAULT[Backup vault stores recovery points]
  VAULT --> LOCK[Vault Lock = immutable WORM]
  ORG[AWS Organizations] --> PLAN
  classDef b fill:#0e2647,stroke:#60a5fa,color:#e7edf3;
  class VAULT b;
  • AWS Backup centralizes protection across many services (EBS, EC2, RDS, EFS, DynamoDB, S3, Storage Gateway and more) and hybrid workloads.
  • Backup plans set schedules, backup window, lifecycle (move to cold storage) and retention; you assign resources by tag or ID.
  • Backups land in a backup vault (KMS-encrypted); Vault Lock makes them immutable (WORM) for compliance.
  • Supports cross-region + cross-account copies and, with Organizations, central backup policies.

Demo

Demo · CLI v2Situation: create a vault + a daily backup plan. Check first: CLI configured. (Running jobs need the AWS Backup service role.)
Create a backup vault
aws backup create-backup-vault --backup-vault-name a4l-vault \
  --query "BackupVaultName"
Expect: a4l-vault created (encrypted with the default KMS key).
Create a daily backup plan
cat > plan.json <<'EOF'
{ "BackupPlanName":"a4l-daily",
  "Rules":[{ "RuleName":"daily-35d",
    "TargetBackupVaultName":"a4l-vault",
    "ScheduleExpression":"cron(0 5 * * ? *)",
    "Lifecycle":{"DeleteAfterDays":35} }] }
EOF
aws backup create-backup-plan --backup-plan file://plan.json \
  --query "BackupPlanId"
Expect: a backup-plan ID. The rule backs up daily at 05:00 UTC and keeps recovery points 35 days.
Fix / Key infoTo actually run backups you assign resources (by tag/ID) to the plan, and AWS Backup uses its service role. For compliance you'd add Vault Lock to make recovery points immutable.
Clean up
PLANID=$(aws backup list-backup-plans \
  --query "BackupPlansList[?BackupPlanName=='a4l-daily'].BackupPlanId" --output text)
aws backup delete-backup-plan --backup-plan-id $PLANID
aws backup delete-backup-vault --backup-vault-name a4l-vault
rm -f plan.json
Expect: plan + empty vault deleted. (A vault with recovery points can't be deleted until they're removed.)
