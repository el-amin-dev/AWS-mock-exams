IAM Identity Policies

Allow / Deny rules attached to users, groups and roles.

Quick Understanding

ConceptMeaningExam relevance
Identity policyJSON rules attached to an identity, saying Allow or Deny.Core of every IAM question.
Statement partsEffect, Action, Resource (+ Condition).Read policies fast.
Priority orderExplicit Deny > Explicit Allow > Default Deny.The #1 IAM rule.
ARNUnique address of a resource.Used in Resource field.

Key Terms

TermSimple meaning
StatementOne rule inside a policy.
EffectAllow or Deny.
ActionWhat is allowed/denied, e.g. s3:GetObject.
ResourceWhat it applies to (an ARN), e.g. a bucket.
Managed policyA reusable policy you attach to many identities.
Inline policyA policy stuck to one identity only (1:1).
Implicit denyIf nothing allows it, it's denied by default.

Explanation

graph TD
  REQ[A request: can I do this?] --> D{Explicit DENY exists?}
  D -->|Yes| NO[DENIED - deny always wins]
  D -->|No| A{Explicit ALLOW exists?}
  A -->|Yes| YES[ALLOWED]
  A -->|No| DEF[DENIED - implicit/default deny]
  classDef r fill:#3a151c,stroke:#fb7185,color:#e7edf3;
  classDef g fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class NO,DEF r; class YES g;
  • A policy has one or more statements: each has Effect, Action, Resource.
  • Decision order: 1) explicit Deny → 2) explicit Allow → 3) default Deny.
  • A single explicit Deny always wins, even with many Allows.
  • Managed policy = reusable (AWS-managed or customer-managed). Inline = tied to one identity.
  • ARN format: arn:aws:service:region:account-id:resource (IAM has blank region — it's global).

Demo

Demo · CLI v2Situation: create user sally, give her S3 read via a managed policy, then prove an explicit Deny on one bucket beats the Allow. Check first: CLI configured as admin.
Create sally + access keys, save them
aws iam create-user --user-name sally
aws iam create-access-key --user-name sally
Expect: JSON with sally's AccessKeyId + SecretAccessKey. Copy both now.
Set up a "sally" CLI profile to test as her
aws configure --profile sally
# paste sally's keys, region us-east-1, output json
Expect: a new profile saved. Test: aws sts get-caller-identity --profile sally shows sally's ARN.
Attach AWS-managed read-only S3 policy
aws iam attach-user-policy --user-name sally \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
Expect: no output. Now aws s3 ls --profile sally works (Allow in effect).
Add an inline explicit DENY on one bucket
cat > deny.json <<'EOF'
{ "Version":"2012-10-17",
  "Statement":[{ "Effect":"Deny","Action":"s3:*",
    "Resource":["arn:aws:s3:::SECRET-BUCKET","arn:aws:s3:::SECRET-BUCKET/*"] }] }
EOF
aws iam put-user-policy --user-name sally \
  --policy-name deny-secret --policy-document file://deny.json
Expect: no output. Sally can list buckets generally, but any action on SECRET-BUCKET is now blocked — the explicit Deny overrides the Allow.
Fix / Key info
  • This proves the rule: explicit Deny > Allow.
  • put-user-policy creates an inline policy; attach-user-policy uses a managed one.
Clean up
aws iam delete-user-policy --user-name sally --policy-name deny-secret
aws iam detach-user-policy --user-name sally \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
KID=$(aws iam list-access-keys --user-name sally --query "AccessKeyMetadata[0].AccessKeyId" --output text)
aws iam delete-access-key --user-name sally --access-key-id $KID
aws iam delete-user --user-name sally
rm -f deny.json
Expect: no output from each = removed. (You can keep the sally profile in your config or remove it manually.)

Build Knowledge

Connects to NOW
  • Groups (Lesson 2) hold these same policies.
  • Roles (Lesson 3) also use permissions policies.
Needed LATER
  • S3 bucket policies, KMS key policies, SCPs — all reuse this Allow/Deny logic.

Exam Pointers

Exam Tip
  • Memorise: Explicit Deny → Explicit Allow → Default (implicit) Deny.
  • Identities start with no access (implicit deny) until allowed.
  • Inline = one identity; managed = reusable across many.
  • An SCP can deny something even if an identity policy allows it (Lesson 7).

Flashcards

Explicit Deny first, then explicit Allow, then default Deny.
Three required parts of a statement?tap
Effect, Action, Resource.
Inline vs managed policy?tap
Inline is tied to one identity; managed is reusable across many.
If nothing allows an action, what happens?tap
It's denied by default (implicit deny).

IAM Groups

Containers for users — not real identities.

Quick Understanding

ConceptMeaningExam relevance
GroupA container that holds IAM users + their shared policies.Admin convenience.
Not an identityNo login, no credentials, can't be a policy principal.Common trap.
No nestingGroups can't contain other groups.Detail question.

Key Terms

TermSimple meaning
PrincipalThe "who" in a resource policy (an identity that acts).
Resource policyA policy attached to a resource (like an S3 bucket) naming who can use it.
Effective permissionsThe total of all policies that apply to a user.

Explanation

graph TD
  G[Group: developers] --> P[Policy attached to group]
  G --> U1[User: sally]
  G --> U2[User: thor]
  U1 --> EFF[Sally's permissions = her own + all her groups]
  GX[Group cannot: log in, hold credentials,
be a resource-policy principal, be nested]
  classDef w fill:#3a151c,stroke:#fb7185,color:#e7edf3;
  class GX w;
  • Add users to a group, attach policies to the group → all members inherit them.
  • A user can be in several groups; their permissions add up.
  • Groups have no credentials and cannot log in.
  • Groups cannot be named as a principal in a resource policy (e.g. a bucket policy).
  • There is no "all users" group and no nesting.

Demo

Demo · CLI v2Situation: move Sally's permissions from her own user into a developers group. Check first: CLI configured as admin.
Recreate sally + a group
aws iam create-user --user-name sally
aws iam create-group --group-name developers
Expect: JSON for the user and the group.
Attach the shared policy to the GROUP, not the user
aws iam attach-group-policy --group-name developers \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
Expect: no output.
Put sally in the group
aws iam add-user-to-group --group-name developers --user-name sally
aws iam list-groups-for-user --user-name sally --query "Groups[].GroupName"
Expect: ["developers"] — sally now inherits the group's S3 read access.
Fix / Key infoBest practice: give permissions through groups, keep user policies empty. Easy to manage at scale.
Clean up
aws iam remove-user-from-group --group-name developers --user-name sally
aws iam detach-group-policy --group-name developers \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
aws iam delete-group --group-name developers
aws iam delete-user --user-name sally
Expect: no output = clean. Detach/remove before deleting.

Build Knowledge

Connects to NOW
  • Holds the identity policies from Lesson 1.
Needed LATER
  • For very large user counts, roles/federation (Lesson 4) replace per-user management.

Exam Pointers

Exam Tip
  • Groups are not identities: no login, no credentials, can't be a principal.
  • Groups can't be nested; there's no built-in "all users" group.
  • Use groups to assign permissions at scale.
TrapIf a bucket policy needs to name "the developers group" as principal — that's impossible. Use a role or list the users, or grant via the group's identity policies instead.

Flashcards

Can you log in as a group?tap
No. Groups have no credentials and aren't identities.
Can a group be a principal in a resource policy?tap
No. Groups can't be referenced as principals.
Can groups be nested?tap
No. A group cannot contain another group.

Mini Quiz

Why can't you put an IAM group in a bucket policy's Principal?
A) Groups aren't identities/principals B) Buckets don't support policies C) Groups are global
A. A group is just a container, not an identity that can act.

IAM Roles – The Tech

A temporary identity you "assume", not log in to.

Quick Understanding

ConceptMeaningExam relevance
RoleAn identity that trusted things assume for a short time.Very high — appears everywhere.
Trust policySays WHO is allowed to assume the role.Half of a role.
Permissions policySays WHAT the role can do once assumed.Other half.
Temporary credsSTS gives short-lived keys that expire.No long-term keys = safer.

Key Terms

TermSimple meaning
Assume a roleTemporarily "become" the role to get its permissions.
STSSecurity Token Service — hands out temporary credentials.
Trust policyThe list of who may assume the role.
Temporary credentialsShort-lived keys (with a session token) that auto-expire.

Explanation

sequenceDiagram
  participant T as Trusted thing (user/service)
  participant R as IAM Role
  participant STS as STS
  T->>R: Can I assume you? (checks Trust policy)
  R->>STS: Yes - issue temp credentials
  STS-->>T: Temp keys + session token (expire soon)
  T->>AWS: Use temp creds = role's permissions
  • A role has two policies: a trust policy (who can assume) + a permissions policy (what it can do).
  • You don't log in to a role — you assume it via STS and get temporary credentials.
  • Those credentials expire, so there are no long-term keys to leak.
  • Roles can be assumed by AWS services, other accounts, or federated users.

Demo

Demo · CLI v2Situation: create a role your own admin can assume to get temporary S3 read access. Check first: CLI configured as admin; know your account ID (aws sts get-caller-identity).
Write a trust policy (who can assume)
ACC=$(aws sts get-caller-identity --query Account --output text)
cat > trust.json <
Expect: a trust.json file allowing your account's identities to assume the role.
Create the role + give it permissions
aws iam create-role --role-name a4l-read \
  --assume-role-policy-document file://trust.json
aws iam attach-role-policy --role-name a4l-read \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
Expect: role JSON (with a role ARN), then no output for the attach.
Assume the role → get temporary credentials
aws sts assume-role \
  --role-arn arn:aws:iam::$ACC:role/a4l-read \
  --role-session-name test
Expect: JSON Credentials with AccessKeyId, SecretAccessKey, SessionToken and an Expiration time — proof they're temporary.
Fix / Key infoThe session token is required for temporary creds. They expire (default ~1 hour), then you assume again.
Clean up
aws iam detach-role-policy --role-name a4l-read \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
aws iam delete-role --role-name a4l-read
rm -f trust.json
Expect: no output = removed.

Build Knowledge

Connects to NOW
  • Permissions policy reuses Lesson 1 logic.
  • Lesson 4 explains WHEN to choose a role.
Needed LATER
  • EC2 instance roles, Lambda execution roles, cross-account access, federation.

Exam Pointers

Exam Tip
  • A role = trust policy (who) + permissions policy (what).
  • You assume a role via STS → get temporary, expiring credentials.
  • Roles have no long-term credentials of their own.
Easy to confuseTrust policy = who may assume. Permissions policy = what the role can do. Both are needed.

Flashcards

A trust policy (who can assume) and a permissions policy (what it can do).
How do you "use" a role?tap
Assume it via STS to get temporary credentials.
Do role credentials expire?tap
Yes — they're temporary and auto-expire.

When to use IAM Roles

The situations where a role is the right answer.

Quick Understanding

ConceptMeaningExam relevance
Service accessAn AWS service needs permissions (EC2, Lambda).Use a role, not stored keys.
Cross-accountLet another account's users act in yours.Role assumed across accounts.
FederationExternal/corporate users get AWS access.Roles + SAML / web identity.
Many usersMore than the IAM user limit (5000).Use roles/federation.

Key Terms

TermSimple meaning
FederationTrusting an outside identity system (e.g. company login, Google).
SAML 2.0A standard used to connect corporate logins to AWS.
Web identityLogin with Google/Facebook/Cognito for app users.
Instance roleA role attached to an EC2 instance so apps on it get permissions.

Explanation

  Q{Need AWS permissions for...?} --> S[An AWS service
EC2 / Lambda] --> R1[Use a ROLE]
  Q --> X[Users in another account] --> R2[Use a cross-account ROLE]
  Q --> F[Corporate / web users] --> R3[Use federation + ROLE]
  Q --> M[More than 5000 users] --> R4[Use federation + ROLE]
  Q --> P[One person, daily admin] --> U[An IAM USER may be fine]
  classDef g fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class R1,R2,R3,R4 g;
  • AWS service needs access (EC2 app → S3, Lambda → DynamoDB) → use a role, never long-term keys.
  • Cross-account: people in account A assume a role in account B.
  • Federation: corporate (SAML) or web/social (Cognito) users assume roles — no IAM user each.
  • IAM users are limited to 5000 per account → big user bases need federation + roles.

  • "Millions of app users" or ">5000 users" → federation + roles, not IAM users.

Service-linked Roles & PassRole

Roles a service owns, and how roles are handed out safely.

Quick Understanding

ConceptMeaningExam relevance
Service-linked role (SLR)A special role pre-defined by, and tied to, one AWS service."Who creates this role?" type Q.
Managed by the serviceThe service decides how it's created/deleted; you can't fully edit it.Differs from normal role.
iam:PassRolePermission to GIVE a role to a service.Stops privilege escalation.

Key Terms

TermSimple meaning
Service-linked roleA role a service uses to act on your behalf, predefined by that service.
PassRoleThe right to attach an existing role to a resource/service.
Privilege escalationGetting more power than you should by misusing permissions.

Explanation

graph TD
  SVC[An AWS service] --> SLR[Service-linked role
predefined + owned by the service]
  SLR --> ACT[Lets the service call other services for you]
  USER[A user] -->|needs iam:PassRole| GIVE[Give a role to EC2/Lambda]
  GIVE --> CHECK[PassRole limits which roles
a user can hand out]
  classDef g fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class SLR g;
  • A service-linked role is created/controlled by the service, with exactly the permissions it needs.
  • You usually can't delete an SLR until the related resources are gone.
  • iam:PassRole = the permission a user needs to assign a role to a service (e.g. attach a role to EC2).
  • PassRole stops a low-power user from handing themselves a powerful role (privilege escalation).

Demo

Demo · CLI v2Situation: create and then remove a service-linked role to see how they work. Check first: CLI configured as admin.
Create a service-linked role (example: AWS Support)
aws iam create-service-linked-role \
  --aws-service-name autoscaling.amazonaws.com
Expect: JSON for a role named like AWSServiceRoleForAutoScaling — created with fixed permissions you didn't choose.
Look at it
aws iam list-roles \
  --query "Roles[?contains(RoleName,'AutoScaling')].RoleName"
Expect: the service-linked role name in the list (path contains aws-service-role).
Fix / Key infoYou can't attach random policies to an SLR; the service defines them. Deletion may be blocked while the service still uses it.
Clean up
aws iam delete-service-linked-role \
  --role-name AWSServiceRoleForAutoScaling
Expect: a DeletionTaskId (deletion is async). If it fails, the service still needs it — that's expected behaviour.

Build Knowledge

Connects to NOW
  • Extends roles (Lesson 3) to service-owned cases.
Needed LATER
  • Auto Scaling, ELB, and many services auto-create SLRs; EC2 instance profiles use PassRole.

Exam Pointers

Exam Tip
  • SLRs are predefined and owned by the service — you can't freely edit them.
  • iam:PassRole is needed to give a role to a service (e.g. EC2). It prevents privilege escalation.
  • If a user "can't launch an instance with a role", they likely lack PassRole.

Flashcards

Who defines a service-linked role's permissions?tap
The AWS service it's linked to (predefined).
What does iam:PassRole allow?tap
Giving (passing) an existing role to a service/resource.
Why does PassRole exist?tap
To stop privilege escalation by limiting which roles a user can hand out.

Service Control Policies (SCPs)

Guardrails that limit what member accounts can do.

Quick Understanding

ConceptMeaningExam relevance
SCPA boundary on what an account is allowed to grant.Very common question.
Never grantsSCPs only set limits; they don't give permissions.The key trap.
Where appliedOrg root, an OU, or a single account.Scope matters.
Mgmt account immuneSCPs don't affect the management account.Remember this.

Key Terms

TermSimple meaning
Guardrail / boundaryA limit on the maximum permissions possible.
Allow listDeny everything, then allow only chosen services.
Deny listAllow all (default), then deny chosen services. (Default style.)
FullAWSAccessThe default SCP that allows everything (deny-list start point).
Effective permissionsWhat an identity can actually do = SCP boundary ∩ identity policy.

Explanation

graph TD
  SCP[SCP boundary - what account CAN allow] --> AND{Both must allow}
  IDP[Identity policy - what user is allowed] --> AND
  AND -->|Yes in both| OK[Action allowed]
  AND -->|Blocked by either| NO[Action denied]
  classDef r fill:#3a151c,stroke:#fb7185,color:#e7edf3;
  classDef g fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class NO r; class OK g;
  • An SCP is a boundary: it sets the maximum an account can grant — it never grants by itself.
  • A user can do something only if both the SCP and the identity policy allow it.
  • Apply SCPs to the org root, an OU, or a single account.
  • Member accounts are affected; the management account is not.
  • Two styles: deny list (default FullAWSAccess + deny some) or allow list (deny all + allow some).

Build Knowledge

Connects to NOW
  • Adds a boundary on top of identity policies (Lesson 1) inside Organizations (Lesson 6).
Needed LATER
  • Control Tower guardrails (Lesson 11) are built from SCPs + Config rules.

Exam Pointers

Exam Tip
  • SCPs limit, never grant. You still need an identity policy to allow.
  • The management account is immune to SCPs.
  • Effective access = SCP ∩ identity policy (allowed in both).
  • Restrict a whole account/OU to certain regions or services → use an SCP.

Flashcards

Do SCPs grant permissions?tap
No. They only set the maximum boundary; identity policies still grant.
Which account ignores SCPs?tap
The management account.
Where can SCPs be attached?tap
Org root, an OU, or an individual account.

Deny list = allow all then deny some (default). Allow list = deny all then allow some.

Using Service Control Policies

Create, apply and test an SCP.

Quick Understanding

ConceptMeaningExam relevance
Enable SCPsSCP policy type must be turned on in the org first.Pre-step often forgotten.
Create + attachMake a policy, attach to root/OU/account.The workflow.
Test effectThe blocked action fails in the member account.See the boundary work.

Key Terms

TermSimple meaning
Policy typeA category of org policy, e.g. SERVICE_CONTROL_POLICY.
TargetWhat the SCP is attached to (root, OU or account ID).

Explanation

graph LR
  E[1. Enable SCP type] --> C[2. Create SCP]
  C --> A[3. Attach to account/OU/root]
  A --> T[4. Test - blocked action fails]
  T --> R[5. Detach + delete to clean up]
  classDef k fill:#0e2647,stroke:#60a5fa,color:#e7edf3;
  class C k;
  • First enable the SCP policy type on the org root.
  • Create the SCP, then attach it to a target.
  • The boundary takes effect on member accounts (not the management account).

Demo

Demo · CLI v2Situation: create a "deny S3" SCP and attach it to your org root, then remove it. Check first: an organization exists (from Lesson 6); run as the management account.
Find the root ID + enable SCPs
ROOT=$(aws organizations list-roots --query "Roots[0].Id" --output text)
aws organizations enable-policy-type \
  --root-id $ROOT --policy-type SERVICE_CONTROL_POLICY
Expect: JSON showing the SCP type now ENABLED on the root.
Create a deny-S3 SCP
cat > scp.json <<'EOF'
{ "Version":"2012-10-17",
  "Statement":[{ "Effect":"Deny","Action":"s3:*","Resource":"*" }] }
EOF
POL=$(aws organizations create-policy --name deny-s3 \
  --type SERVICE_CONTROL_POLICY --description "block S3" \
  --content file://scp.json --query "Policy.PolicySummary.Id" --output text)
echo $POL
Expect: a policy ID like p-xxxx in $POL.
Attach it (here: to the root)
aws organizations attach-policy --policy-id $POL --target-id $ROOT
Expect: no output. In a member account, any S3 action now fails — even with an S3 Allow. (Your management account is unaffected.)
Fix / Key infoAttaching "deny S3" to the root would hit all member accounts. In real life, attach to a specific OU/account to limit blast radius.
Clean up
aws organizations detach-policy --policy-id $POL --target-id $ROOT
aws organizations delete-policy --policy-id $POL
rm -f scp.json
Expect: no output = removed. (You can leave the SCP type enabled; it has no effect without attached policies.)

CloudWatch Logs

Collect, store and monitor log data.

Quick Understanding

ConceptMeaningExam relevance
CloudWatch LogsA place to send and keep log data, then search/alarm on it.Common log target.
Public + regionalReachable over the internet; data sits in a region.Works from on-prem too.
StructureLog group → log stream → log events.Know the hierarchy.
Metric filterTurn matching log lines into a metric → alarm.Useful pattern.

Key Terms

TermSimple meaning
Log groupA container for related logs (e.g. one app).
Log streamA sequence of log lines from one source (e.g. one instance).
Log eventOne log line with a timestamp.
Metric filterA rule that counts/extracts values from logs into a CloudWatch metric.

Explanation

graph TD
  SRC[Sources: EC2 agent, Lambda, VPC Flow Logs, CloudTrail, on-prem] --> LG[Log Group]
  LG --> LS1[Log Stream - instance 1]
  LG --> LS2[Log Stream - instance 2]
  LS1 --> EV[Log Events]
  LG --> MF[Metric Filter] --> AL[CloudWatch Alarm]
  classDef k fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class LG k;
  • Hierarchy: log group → log streams → log events.
  • It's a public, regional service — usable from EC2 (via agent), Lambda, VPC Flow Logs, even on-prem.
  • Metric filters turn log patterns into metrics you can alarm on.

Demo

Demo · CLI v2Situation: create a log group, push a log line, read it back, then delete. Check first: CLI configured; pick a region (us-east-1).
Create a log group + stream
aws logs create-log-group --log-group-name a4l-demo
aws logs create-log-stream --log-group-name a4l-demo --log-stream-name s1
Expect: no output = created.
Put a log event
aws logs put-log-events --log-group-name a4l-demo --log-stream-name s1 \
  --log-events timestamp=$(($(date +%s)*1000)),message="hello cats"
Expect: JSON with a nextSequenceToken = the event was stored.
Read it back
aws logs get-log-events --log-group-name a4l-demo --log-stream-name s1 \
  --query "events[].message"
Expect: ["hello cats"].
Fix / Key infoBy default logs are kept forever (you pay for storage). Set retention to auto-delete: aws logs put-retention-policy --log-group-name a4l-demo --retention-in-days 7.
Clean up
aws logs delete-log-group --log-group-name a4l-demo
Expect: no output = group and its streams deleted. No more storage cost.

Connects to NOW

  • Metric filters reuse CloudWatch alarms (Section 002).
Needed LATER
  • VPC Flow Logs, Lambda logs, the CloudWatch Agent, subscription filters.

Exam Pointers

Exam Tip
  • Hierarchy: group → stream → event.
  • EC2 needs the CloudWatch Agent to send OS logs (not automatic).
  • Use a metric filter to alarm on a pattern in logs (e.g. "ERROR" count).
  • Set retention or logs are kept (and billed) forever.

CloudTrail

Records who did what (API calls and account events).

Quick Understanding

ConceptMeaningExam relevance
CloudTrailLogs API calls + account activity for audit."Who deleted X?" → CloudTrail.
Event historyLast 90 days of management events, free, always on.Free vs trail trap.
TrailStores events long-term to S3 (and/or CloudWatch Logs).For >90 days.
Event typesManagement (default) vs Data (off by default, costs).Common detail.

Key Terms

TermSimple meaning
Management eventControl actions (create/delete a resource). Logged by default.
Data eventObject-level actions (S3 GetObject, Lambda invoke). Off by default, billed.
TrailA configuration that saves events to S3 / CloudWatch Logs.
Organizational trailOne trail logging every account in the org.
Global service eventsIAM/STS/CloudFront events, logged in us-east-1.

Explanation

graph TD
  API[Any API call / account event] --> CT[CloudTrail]
  CT --> HIST[90-day Event History
free, mgmt events only]
  CT --> TRAIL[Trail]
  TRAIL --> S3[S3 - long-term store]
  TRAIL --> CWL[CloudWatch Logs - search/alarm]
  classDef k fill:#0e2647,stroke:#60a5fa,color:#e7edf3;
  class CT k;
  • Free 90-day Event History is always on (management events only).
  • For longer storage or data events, create a trail → sends to S3 (and optionally CloudWatch Logs).
  • Management events log by default; data events are off by default and cost extra.
  • Global service events (IAM/STS/CloudFront) appear in us-east-1.
  • An organizational trail captures all accounts in the org at once.
  • CloudTrail is not real-time (events arrive within ~15 minutes).

Demo

Demo · CLI v2Situation: create a trail that logs to S3 for Animals4life, then remove it. Check first: CLI configured. Cost note: a trail to S3 is cheap but not zero (~$0.01 per 2000 requests; CloudTrail is noisy) — delete it after.
Create an S3 bucket + give CloudTrail permission
ACC=$(aws sts get-caller-identity --query Account --output text)
BUCKET=a4l-trail-$RANDOM$RANDOM
aws s3 mb s3://$BUCKET --region us-east-1
cat > bp.json <
Expect: bucket created and policy set (no output from put-bucket-policy).
Create + start the trail (all regions, global events)
aws cloudtrail create-trail --name a4l-trail \
  --s3-bucket-name $BUCKET \
  --is-multi-region-trail --include-global-service-events
aws cloudtrail start-logging --name a4l-trail
Expect: trail JSON (with its ARN); start-logging gives no output. Logs appear in S3 within ~15 min.
Fix / Key info
  • Multi-region trail = captures every region in one trail (recommended).
  • Data events (e.g. S3 object reads) are not included unless you add them — and they cost more.
Clean up
aws cloudtrail stop-logging --name a4l-trail
aws cloudtrail delete-trail --name a4l-trail
aws s3 rm s3://$BUCKET --recursive
aws s3 rb s3://$BUCKET
rm -f bp.json
Expect: trail deleted, bucket emptied + removed. No ongoing cost.

Exam Tip
