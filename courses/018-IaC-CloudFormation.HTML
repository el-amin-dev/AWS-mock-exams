CloudFormation vs Terraform

Picking an IaC tool (optional).

Quick Understanding

ToolWhatExam relevance
CloudFormationAWS-native IaC, free, YAML/JSON.AWS-only; managed state.
Terraform3rd-party, multi-cloud, HCL.You manage state.

Key Terms

TermSimple meaning
IaCInfrastructure as Code — define infra in text files.
StateA record of what's been deployed.

Explanation

  • IaC = define infrastructure in code so it's repeatable and version-controlled.
  • CloudFormation: AWS-native, free (pay only for resources), YAML/JSON, AWS manages the state (stacks) — best for AWS-only.
  • Terraform: third-party (HashiCorp), multi-cloud, HCL language, you manage a state file — best when you span multiple clouds.

Physical & Logical Resources

Template = WHAT, stack = the real things.

Quick Understanding

ConceptMeaningExam relevance
Logical resourceThe WHAT, declared in the template.You write this.
Physical resourceThe real thing CFN creates.One per logical.
StackAll resources from one template.Create/update/delete together.
PortabilityAvoid hardcoded values.Reuse anywhere.

Key Terms

TermSimple meaning
TemplateA YAML/JSON file describing resources.
StackA deployed instance of a template.
Non-portableHardcoded so it only works once/one region.

Explanation

graph TD
  TPL[Template: logical resources - WHAT] --> STACK[Stack]
  STACK --> PHYS[Physical resources - real AWS things]
  TPL -->|update template| STACK
  STACK -->|create/update/delete| PHYS
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class STACK a;
  • A template declares logical resources (the WHAT); CloudFormation works out the HOW.
  • A stack creates one physical resource per logical resource, and updates/deletes them as the template changes.
  • Resources is the only required section. Non-portable templates hardcode values (AMI IDs, names) — they break in other regions; use parameters/mappings to make them portable.

Demo

Demo · CLI v2 (deploying an empty S3 bucket is free)Situation: deploy your first stack. Check first: CLI configured. (Keep this stack for L17.)
Write a tiny template + validate it
cat > tpl.yaml << 'EOF'
AWSTemplateFormatVersion: '2010-09-09'
Description: A4L first stack
Resources:
  A4LBucket:
    Type: AWS::S3::Bucket
EOF
aws cloudformation validate-template --template-body file://tpl.yaml
Expect: the template validates (no errors).
Create the stack
aws cloudformation create-stack --stack-name a4l-demo \
  --template-body file://tpl.yaml
aws cloudformation wait stack-create-complete --stack-name a4l-demo
aws cloudformation describe-stacks --stack-name a4l-demo \
  --query "Stacks[0].StackStatus" --output text
Expect: CREATE_COMPLETE + a physical S3 bucket created for the logical A4LBucket.
Fix / Key infoThe logical name (A4LBucket) is what you reference in the template; the physical name is auto-generated. Keep this stack — L17 reuses it.
Clean up (or keep for L17)
aws cloudformation delete-stack --stack-name a4l-demo
Expect: deleting the stack deletes the bucket (default behaviour).

Connects to NOW

Needed LATER
  • All IaC deployments.

Exam Tip
  • Logical resource (template) → physical resource (stack creates it).
  • Resources is the only required template section.
  • Hardcoded = non-portable; use parameters/mappings/pseudo params for portability.

Key Terms

TermSimple meaning
ParameterA value passed in when deploying.

Explanation

PortabilityPick values per region/env.Reusable templates.

Key Terms

TermSimple meaning
MappingA fixed lookup table of keys → values.
FindInMapThe function that reads a mapping.

Explanation

  • Mappings match a key to a set of named values — classic use is region → AMI ID, so one template works in many regions.
  • Read with !FindInMap [MapName, TopKey, SecondKey].
  • Modern alternative for AMIs: SSM public parameters. [added for exam]
Mappings:
  RegionAMI:
    us-east-1: { AMI: ami-1111 }
    eu-west-1: { AMI: ami-2222 }
Resources:
  Inst:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: !FindInMap [RegionAMI, !Ref "AWS::Region", AMI]

Key Terms

TermSimple meaning
ConditionA named true/false test.
Fn::EqualsChecks if two values are equal.

Explanation

  • Conditions use logic (Fn::Equals/And/Or/Not) on parameters/pseudo params; attach Condition: to a resource so it's created only when true (e.g. prod-only resources).
Parameters:
  EnvType: { Type: String, AllowedValues: [test, prod] }
Conditions:
  IsProd: !Equals [!Ref EnvType, prod]
Resources:
  ProdOnlyBucket:
    Type: AWS::S3::Bucket

Wait Conditions & cfn-signal

Wait until EC2 is really ready.

Quick Understanding

ConceptMeaningExam relevance
CreationPolicyWait for N signals or timeout.Preferred method.
cfn-signalScript that signals success/failure.Run in userdata.
WaitConditionOlder signalling method.Legacy.

Key Terms

TermSimple meaning
SignalA message saying "I'm ready / I failed".
CreationPolicyTells CFN to wait for signals before "complete".

Explanation

graph TD
  EC2[EC2 launches] --> BOOT[userdata bootstraps app]
  BOOT --> SIG[cfn-signal: success/failure]
  SIG --> CFN{CreationPolicy: got N signals?}
  CFN -->|yes| OK[CREATE_COMPLETE]
  CFN -->|timeout| FAIL[Failed]
  classDef a fill:#0e2647,stroke:#60a5fa,color:#e7edf3;
  class SIG a;
  • A CreationPolicy makes CloudFormation wait for a set number of success signals (or a timeout) before marking a resource CREATE_COMPLETE.
  • The cfn-signal helper (run in userdata) sends success/failure after bootstrap — so the stack isn't "done" until the app actually works.
  • WaitCondition + WaitConditionHandle is the older method; CreationPolicy is preferred.

ChangeSets

Preview changes before applying.

Quick Understanding

ConceptMeaningExam relevance
PreviewSee add/modify/replace/delete.Avoid surprises.
Replacement riskSome changes recreate a resource.Downtime/data loss.
Execute or discardApply only when ready.Safe updates.

Key Terms

TermSimple meaning
Change setA preview of what an update would do.
ReplacementDeleting + recreating a resource.

Explanation

  • A change set shows what an update would add/modify/replace/delete — especially whether a resource will be replaced (which can mean downtime/data loss).
  • CloudFormation only applies changes when you execute the change set — otherwise you can discard it.

Demo

Demo · CLI v2 (uses the L2 stack)Situation: preview a change before applying. Check first: the a4l-demo stack from L2 exists.
Edit the template (add a tag), create a change set
cat > tpl2.yaml << 'EOF'
AWSTemplateFormatVersion: '2010-09-09'
Resources:
  A4LBucket:
    Type: AWS::S3::Bucket
    Properties:
      Tags:
        - { Key: env, Value: demo }
EOF
aws cloudformation create-change-set --stack-name a4l-demo \
  --change-set-name a4l-cs --template-body file://tpl2.yaml
Expect: a change set is created (status CREATE_COMPLETE when ready).
Preview what will change
aws cloudformation describe-change-set --stack-name a4l-demo \
  --change-set-name a4l-cs \
  --query "Changes[].ResourceChange.{Action:Action,Res:LogicalResourceId,Replace:Replacement}"
Expect: a Modify on A4LBucket with Replacement: False (a tag is a safe change).
Execute it (apply the change)
aws cloudformation execute-change-set --stack-name a4l-demo \
  --change-set-name a4l-cs
Expect: the stack updates; the bucket gets the tag.
Fix / Key infoChange sets warn you when a change forces a Replacement (recreate) — critical for stateful resources.
Clean up
aws cloudformation delete-stack --stack-name a4l-demo
rm -f tpl.yaml tpl2.yaml
Expect: stack + bucket deleted (this removes the L2 stack too).
