Bootstrapping with User Data

Run a script automatically at first launch.

Quick Understanding

ConceptMeaningExam relevance
BootstrappingAuto-configure an instance right after launch.Automation basics.
User dataA script passed via metadata, run by the OS at boot.Runs once, as root.
Not secretReadable from metadata — no passwords/keys.Security trap.
No waitStatus checks pass before user-data finishes.Why we need cfn-signal.

Key Terms

TermSimple meaning
User dataText/script you attach to an instance; the OS runs it at boot.
cloud-initThe boot tool that runs user data on Linux.
IdempotentSafe to run again with the same result (user data is NOT, by default).

Explanation

graph TD
  LAUNCH[Launch instance + user data] --> META[Stored in metadata /latest/user-data]
  META --> CLOUDINIT[cloud-init runs the script as root]
  CLOUDINIT --> APP[Install + configure app]
  NOTE[Runs ONCE on first boot · max 16 KB · not encrypted]
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class CLOUDINIT a;
  • User data is run by cloud-init as root, once on first boot. AWS treats it as opaque (just runs it).
  • It's not encrypted and readable from metadata → never put secrets in it.
  • Limit 16 KB (raw). For bigger setups, download a script or use config tools.
  • The instance reaches running / status-OK before user data finishes — so don't assume it's "ready". (L2 fixes this with cfn-signal.)

Demo

Demo · CLI v2Situation: bootstrap an instance with user data, then diagnose it. Check first: CLI configured; a subnet (default VPC fine). Note: t3.micro free-tier — verify your region.
Launch with a user-data script
cat > ud.sh <<'EOF'
#!/bin/bash
dnf install -y httpd cowsay
echo "$(cowsay Hello from Animals4life)" > /var/www/html/index.html
systemctl enable --now httpd
EOF
AMI=$(aws ssm get-parameters \
  --names /aws/service/ami-amazon-linux-latest/al2023-ami-kernel-default-x86_64 \
  --query "Parameters[0].Value" --output text)
INST=$(aws ec2 run-instances --image-id $AMI --instance-type t3.micro \
  --user-data file://ud.sh --query "Instances[0].InstanceId" --output text)
echo $INST
Expect: an instance ID; on boot it installs Apache + cowsay and serves a page.
Diagnose: read the user data back via metadata (run ON the instance)
TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 300")
curl -s -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/user-data
Expect: your script text printed back — proof it was passed in (and that it's not secret).
Diagnose: check the boot log (run ON the instance)
sudo cat /var/log/cloud-init-output.log | tail -n 30
Expect: output from your install commands — this is where you debug failed bootstraps.
Fix / Key infoA CloudFormation template passes the same script in its UserData property (base64). The big limit: the instance is "running" before the script finishes — see L2 for the proper wait/signal.
Clean up
aws ec2 terminate-instances --instance-ids $INST
rm -f ud.sh
Expect: instance terminating; root volume removed by default.

  • Instance shows running before user data completes → can't rely on it being "ready".
  • Debug at /var/log/cloud-init-output.log.

Enhanced Bootstrapping with CFN-INIT

Desired-state config + wait for a success signal.

Quick Understanding

ConceptMeaningExam relevance
cfn-initApplies desired state (packages/files/services) from template metadata.Vs procedural user data.
cfn-signalReports success/failure back to CloudFormation.Confirms readiness.
CreationPolicyMakes CFN wait for a signal before CREATE_COMPLETE.The "wait state".

Key Terms

TermSimple meaning
Desired stateYou declare the end result; the tool makes it happen.
CreationPolicyA rule: "wait for N success signals or time out".
Metadata (Init)The AWS::CloudFormation::Init block describing the config.

Explanation

sequenceDiagram
  participant CFN as CloudFormation
  participant EC2 as Instance
  CFN->>EC2: Launch (user data calls cfn-init)
  EC2->>EC2: cfn-init applies desired state
  EC2->>CFN: cfn-signal (success/failure)
  Note over CFN: CreationPolicy waits for signal
  CFN->>CFN: Mark CREATE_COMPLETE only after success
  • cfn-init reads AWS::CloudFormation::Init and brings the instance to a desired state (packages, files, users, services) — more robust + re-runnable than a plain script.
  • cfn-signal tells CloudFormation whether the setup succeeded.
  • CreationPolicy holds the resource in a wait state until the signal arrives (or it times out) — so the stack only completes when the app is truly ready.

Demo

Demo · CLI v2 + CloudFormationSituation: deploy a stack that uses cfn-init + cfn-signal + a CreationPolicy. Check first: CLI configured; default VPC.
Write a small template
cat > cfninit.yaml <<'EOF'
Parameters:
  LatestAmi:
    Type: AWS::SSM::Parameter::Value<AWS::EC2::Image::Id>
    Default: /aws/service/ami-amazon-linux-latest/al2023-ami-kernel-default-x86_64
Resources:
  Web:
    Type: AWS::EC2::Instance
    CreationPolicy:
      ResourceSignal:
        Count: 1
        Timeout: PT10M
    Metadata:
      AWS::CloudFormation::Init:
        config:
          packages:
            yum: { httpd: [] }
          files:
            /var/www/html/index.html:
              content: "Hello from cfn-init\n"
          services:
            sysvinit:
              httpd: { enabled: true, ensureRunning: true }
    Properties:
      InstanceType: t3.micro
      ImageId: !Ref LatestAmi
      UserData:
        Fn::Base64: !Sub |
          #!/bin/bash
          dnf install -y aws-cfn-bootstrap
          /opt/aws/bin/cfn-init -v --stack ${AWS::StackName} --resource Web --region ${AWS::Region}
          /opt/aws/bin/cfn-signal -e $? --stack ${AWS::StackName} --resource Web --region ${AWS::Region}
EOF
echo "template written"
Expect: template written. (The < > become < > in your editor — they're the SSM parameter type.)
Deploy and watch it WAIT for the signal
aws cloudformation create-stack --stack-name a4l-cfninit \
  --template-body file://cfninit.yaml
aws cloudformation wait stack-create-complete --stack-name a4l-cfninit
echo "stack complete (only after cfn-signal success)"
Expect: the wait holds until cfn-signal reports success, then completes — not before the app is ready.
Fix / Key infoIf the instance fails to signal within the timeout, the stack fails and rolls back — exactly what you want for reliable bootstrapping.
Clean up
aws cloudformation delete-stack --stack-name a4l-cfninit
aws cloudformation wait stack-delete-complete --stack-name a4l-cfninit
rm -f cfninit.yaml
Expect: stack + its instance deleted.

Exam Tip

  • Use these when the stack must not complete until the app is truly ready.
  • cfn-init is re-runnable; plain user data is a one-shot script.

EC2 Instance Roles & Profile

Give apps permissions without stored keys.

Quick Understanding

ConceptMeaningExam relevance
Instance roleAn IAM role the instance assumes.Best-practice creds.
Instance profileWrapper that attaches the role to the instance.The actual link.
Temp credentialsDelivered + auto-rotated via metadata.No keys to manage.

Key Terms

TermSimple meaning
Instance profileA container holding one role; it's what you attach to the instance.
STSThe service that issues temporary credentials.
Auto-rotateEC2 + STS refresh the credentials before they expire.

Explanation

graph TD
  ROLE[IAM Role + policy] --> PROFILE[Instance Profile wraps the role]
  PROFILE --> INST[Attached to instance]
  INST --> META[Temp creds in metadata: iam/security-credentials/ROLE]
  META --> SDK[CLI/SDK use them automatically + auto-rotate]
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class PROFILE a;
  • An instance role lets the app call AWS using temporary credentials — no access keys on disk.
  • The role is attached through an instance profile (the console makes it for you; with the CLI you create it).
  • Credentials appear in metadata at iam/security-credentials/<role> and are auto-rotated by EC2 + STS.
  • The CLI/SDK on the instance pick them up automatically.

Demo

Demo · CLI v2Situation: create a role that allows S3 read, attach to an instance, read creds from metadata. Check first: a running instance ID in $INST.
Create the role with a trust policy for EC2
cat > trust.json <<'EOF'
{ "Version":"2012-10-17","Statement":[{"Effect":"Allow",
  "Principal":{"Service":"ec2.amazonaws.com"},"Action":"sts:AssumeRole"}]}
EOF
aws iam create-role --role-name a4l-ec2-s3ro \
  --assume-role-policy-document file://trust.json
aws iam attach-role-policy --role-name a4l-ec2-s3ro \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
Expect: role created; S3 read-only policy attached.
Create an instance profile + add the role + attach
aws iam create-instance-profile --instance-profile-name a4l-ec2-s3ro
aws iam add-role-to-instance-profile \
  --instance-profile-name a4l-ec2-s3ro --role-name a4l-ec2-s3ro
sleep 10
aws ec2 associate-iam-instance-profile --instance-id $INST \
  --iam-instance-profile Name=a4l-ec2-s3ro
Expect: an association ID — the instance now has the role.
On the instance: see the temporary credentials + test
TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 300")
curl -s -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/iam/security-credentials/a4l-ec2-s3ro
aws s3 ls   # works with no keys configured
Expect: temporary AccessKeyId/SecretAccessKey/Token JSON, and aws s3 ls succeeds using them.
Fix / Key infoThe role holds the permissions; the instance profile is the link that puts the role on the instance. Never store long-term keys on an instance.
Clean up
ASSOC=$(aws ec2 describe-iam-instance-profile-associations \
  --filters Name=instance-id,Values=$INST \
  --query "IamInstanceProfileAssociations[0].AssociationId" --output text)
aws ec2 disassociate-iam-instance-profile --association-id $ASSOC
aws iam remove-role-from-instance-profile \
  --instance-profile-name a4l-ec2-s3ro --role-name a4l-ec2-s3ro
aws iam delete-instance-profile --instance-profile-name a4l-ec2-s3ro
aws iam detach-role-policy --role-name a4l-ec2-s3ro \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
aws iam delete-role --role-name a4l-ec2-s3ro
rm -f trust.json
Expect: all role/profile resources removed.

SSM Parameter Store

Store config and secrets, the AWS way.

Quick Understanding

ConceptMeaningExam relevance
Parameter StoreStores config values + secrets in Systems Manager.Config/secret store.
TypesString, StringList, SecureString (KMS-encrypted).SecureString = secrets.
PathsHierarchical keys like /a4l/db/password.Fetch by path.
IAM + KMSAccess via IAM; encrypt with KMS; versioned.Secure by design.

Key Terms

TermSimple meaning
SecureStringAn encrypted parameter (uses KMS).
PathA folder-like name to group parameters.
VersionEach change keeps a numbered history.

Explanation

graph TD
  APP[App / instance role] -->|GetParameter| PS[(Parameter Store)]
  PS --> STR[String / StringList]
  PS --> SEC[SecureString - KMS encrypted]
  PS --> PATH[Fetch many by /path]
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class PS a;
  • Stores String, StringList, and SecureString (encrypted with KMS) values.
  • Access is controlled by IAM; values are versioned; reachable from anywhere via the public endpoint.
  • Use paths (e.g. /a4l/db/...) to organise and fetch groups of parameters.
  • Integrates natively with many services (CloudFormation, Lambda, EC2, the CW agent in L5).

Demo

Demo · CLI v2 (standard params are free)Situation: store and read config + a secret. Check first: CLI configured.
Put a String and a SecureString
aws ssm put-parameter --name /a4l/web/dbhost \
  --value "db.internal.a4l" --type String
aws ssm put-parameter --name /a4l/web/dbpassword \
  --value "SuperSecret123" --type SecureString
Expect: each returns Version: 1. SecureString is encrypted with the default KMS key.
Read them back (decrypt the secret)
aws ssm get-parameter --name /a4l/web/dbhost --query "Parameter.Value" --output text
aws ssm get-parameter --name /a4l/web/dbpassword --with-decryption \
  --query "Parameter.Value" --output text
Expect: the host in plain text; the password only decrypts with --with-decryption (and KMS permission).
Fetch everything under a path
aws ssm get-parameters-by-path --path /a4l/web --recursive --with-decryption \
  --query "Parameters[].{Name:Name,Value:Value}"
Expect: both parameters listed — handy for an app to pull all its config at once.
Fix / Key infoAn EC2 app reads these using its instance role (L3); SecureString also needs KMS decrypt permission. For heavy secret rotation, Secrets Manager is the dedicated option. [added for exam]
Clean up
aws ssm delete-parameters --names /a4l/web/dbhost /a4l/web/dbpassword
Expect: both names listed under DeletedParameters.

System & Application Logging (CloudWatch Agent)

Get in-OS metrics + logs into CloudWatch.

Quick Understanding

ConceptMeaningExam relevance
Default CW gapDefault metrics are host-level only — no memory/disk/logs.Why the agent exists.
CloudWatch agentSoftware on the instance that sends OS metrics + logs.Logs → CW Logs.
Needs a roleInstance role grants put-metrics/logs + read config.Ties to L3/L4.

Key Terms

TermSimple meaning
CloudWatch LogsWhere log files are stored + searched.
Agent configA file saying which metrics/logs to collect (can live in Parameter Store).
Custom metricA metric AWS can't see by default, e.g. memory used.

Explanation

graph TD
  INST[Instance + CloudWatch agent] -->|metrics| CW[CloudWatch Metrics]
  INST -->|log files| CWL[CloudWatch Logs]
  ROLE[Instance role: CloudWatchAgentServerPolicy] --> INST
  PS[(Parameter Store: agent config)] --> INST
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  class CW,CWL a;
  • By default CloudWatch sees only host-level metrics (CPU, network). It can't see in-OS things like memory, disk space, or log files.
  • The CloudWatch agent fills that gap: it sends OS metrics and log files to CloudWatch + CloudWatch Logs.
  • It needs an instance role (e.g. CloudWatchAgentServerPolicy) to write metrics/logs and to read its config from Parameter Store.
  • Example logs to capture: /var/log/secure, /var/log/httpd/access_log, /var/log/httpd/error_log.

Demo

Demo · role (CLI) + agent (on instance)Situation: let an instance ship 3 log files to CloudWatch Logs. Check first: a running instance in $INST; CW Logs charges are tiny but real.
Give the instance the agent role (CLI)
cat > trust.json <<'EOF'
{ "Version":"2012-10-17","Statement":[{"Effect":"Allow",
  "Principal":{"Service":"ec2.amazonaws.com"},"Action":"sts:AssumeRole"}]}
EOF
aws iam create-role --role-name a4l-cwagent --assume-role-policy-document file://trust.json
aws iam attach-role-policy --role-name a4l-cwagent \
  --policy-arn arn:aws:iam::aws:policy/CloudWatchAgentServerPolicy
aws iam attach-role-policy --role-name a4l-cwagent \
  --policy-arn arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore
aws iam create-instance-profile --instance-profile-name a4l-cwagent
aws iam add-role-to-instance-profile --instance-profile-name a4l-cwagent --role-name a4l-cwagent
sleep 10
aws ec2 associate-iam-instance-profile --instance-id $INST \
  --iam-instance-profile Name=a4l-cwagent
Expect: role attached; the agent can now put logs/metrics + read its config.
On the instance: install + configure the agent
sudo dnf install -y amazon-cloudwatch-agent
# run the wizard to pick the 3 log files, OR drop a config and start:
sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \
  -a fetch-config -m ec2 \
  -c ssm:AmazonCloudWatch-a4l -s
Expect: agent running; it reads the config named AmazonCloudWatch-a4l from Parameter Store and starts shipping logs.
Confirm log groups exist (CLI)
aws logs describe-log-groups --query "logGroups[].logGroupName"
Expect: log groups for the captured files appear after a minute.
Fix / Key infoThe agent config can be stored in Parameter Store so many instances share it. The role needs both put-logs/metrics and read-config permissions.
Clean up
# delete any created log groups (repeat per name), then remove the role
aws logs delete-log-group --log-group-name /var/log/secure 2>/dev/null
ASSOC=$(aws ec2 describe-iam-instance-profile-associations \
  --filters Name=instance-id,Values=$INST \
  --query "IamInstanceProfileAssociations[0].AssociationId" --output text)
aws ec2 disassociate-iam-instance-profile --association-id $ASSOC
aws iam remove-role-from-instance-profile --instance-profile-name a4l-cwagent --role-name a4l-cwagent
aws iam delete-instance-profile --instance-profile-name a4l-cwagent
aws iam detach-role-policy --role-name a4l-cwagent --policy-arn arn:aws:iam::aws:policy/CloudWatchAgentServerPolicy
aws iam detach-role-policy --role-name a4l-cwagent --policy-arn arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore
aws iam delete-role --role-name a4l-cwagent ; rm -f trust.json
Expect: log groups + role removed (stops CW Logs storage charges).

EC2 Placement Groups

Control where instances sit on hardware.

Quick Understanding

TypeGoalExam relevance
ClusterPack close together for top performance.HPC, low latency.
SpreadKeep apart on distinct hardware.Max resilience (max 7/AZ).
PartitionGroups on separate racks; topology aware.Big distributed apps.

Key Terms

TermSimple meaning
Placement groupA rule for how AWS places your instances on hardware.
PartitionA set of racks isolated from other partitions' racks.
Topology awareThe app knows which partition each node is in.

Explanation

graph TD
  CL[Cluster: same rack, 1 AZ - fastest network] --> CLR[Risk: rack failure hits all]
  SP[Spread: distinct hardware - max 7 per AZ] --> SPR[Best for few critical instances]
  PA[Partition: up to 7 partitions/AZ, many instances] --> PAR[HDFS, Cassandra, Kafka]
  classDef a fill:#0f3b39,stroke:#2dd4bf,color:#e7edf3;
  classDef b fill:#0e2647,stroke:#60a5fa,color:#e7edf3;
  class CL a; class SP,PA b;
  • Cluster: instances packed in one AZ, same rack → highest network performance + lowest latency (HPC). Downside: a rack failure can take them all.
  • Spread: instances on distinct hardware → maximum resilience for a small number of critical instances (max 7 per AZ).
  • Partition: up to 7 partitions per AZ, each on separate racks, with many instances — for large distributed/replicated systems that are topology aware.

Demo

Demo · CLI v2 (creating groups is free)Situation: create each placement-group type and inspect them. Check first: CLI configured.
Create the three types
aws ec2 create-placement-group --group-name a4l-cluster --strategy cluster
aws ec2 create-placement-group --group-name a4l-spread --strategy spread
aws ec2 create-placement-group --group-name a4l-part \
  --strategy partition --partition-count 3
Expect: no errors. Creating groups is free (you only pay for instances you launch into them).
List them
aws ec2 describe-placement-groups \
  --query "PlacementGroups[].{Name:GroupName,Strategy:Strategy}"
Expect: the three groups with strategies cluster / spread / partition.
Fix / Key infoYou'd launch instances with --placement GroupName=a4l-cluster. Spread caps at 7 instances per AZ; partition allows many but limits partitions.
Clean up
aws ec2 delete-placement-group --group-name a4l-cluster
aws ec2 delete-placement-group --group-name a4l-spread
aws ec2 delete-placement-group --group-name a4l-part
Expect: all three removed (they were free anyway).

Faster network + dedicated storage bandwidth.

Quick Understanding

ConceptMeaningExam relevance
Enhanced networkingSR-IOV: card presents virtual NICs to the instance.Higher net performance.
BenefitsLower host CPU, higher throughput, low/consistent latency.No extra cost.

Key Terms

TermSimple meaning
SR-IOVHardware feature: one card acts as many virtual cards.
ENAElastic Network Adapter — the modern enhanced-networking driver.
EBS OptimizedA separate network lane just for EBS traffic.

Explanation

graph TD
  CARD[Physical NIC + SR-IOV] --> VNIC[Virtual NICs straight to instances]
  VNIC --> PERF[Lower CPU, higher throughput, consistent latency]
  EBSOPT[EBS Optimized: dedicated storage bandwidth] --> SEP[Storage traffic separate from data traffic]
  classDef b fill:#0e2647,stroke:#60a5fa,color:#e7edf3;
  class VNIC,EBSOPT b;
  • Enhanced networking is AWS's SR-IOV: the physical card gives instances virtual NICs directly → lower host CPU, higher throughput, lower + more consistent latency. No extra charge; on most modern types via ENA.
  • EBS Optimized gives dedicated bandwidth for EBS storage traffic, separate from normal data networking → steadier EBS performance. Most modern instances are EBS-optimized by default.

Demo

Demo · CLI v2 (read + toggle)Situation: check ENA + EBS-optimized state. Check first: an instance ID in $INST.
Check enhanced networking (ENA) + EBS-optimized
aws ec2 describe-instances --instance-ids $INST \
  --query "Reservations[0].Instances[0].{ENA:EnaSupport,EbsOpt:EbsOptimized}"
Expect: ENA: true on modern types; EbsOpt true/false depending on type.
(Optional) enable EBS optimization (instance must be stopped)
aws ec2 stop-instances --instance-ids $INST
aws ec2 wait instance-stopped --instance-ids $INST
aws ec2 modify-instance-attribute --instance-id $INST --ebs-optimized
aws ec2 start-instances --instance-ids $INST
Expect: attribute set (if the type supports toggling; many are on by default and can't be turned off).
Fix / Key infoBoth features are about performance, not security, and mostly come on by default on current instance types — you rarely configure them manually.
